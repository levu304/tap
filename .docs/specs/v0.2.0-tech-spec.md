## 1. Overview

Tap v0.2.0 — codename **"Polyglot"** — is the second milestone of the Tap CDC platform. Where v0.1.0 ("Foundation") proved the core engine with a single database (Postgres), a single language (TypeScript), and a single transport (HTTP/2 SSE), v0.2.0 expands across all three axes simultaneously.

This release introduces four major capabilities:

1. **MySQL support**: A new binlog adapter using ROW-based replication brings MySQL and MariaDB databases into the capture pipeline alongside Postgres.
2. **Python SDK**: A `pyo3` + `maturin`-powered native extension delivers the same embedded capture experience to Python/asyncio applications that v0.1.0 gave to Node.js/TypeScript, including Pydantic model generation from live schema.
3. **Sidecar mode**: A standalone `tap-sidecar` binary exposes a REST + SSE HTTP API on `localhost:9911`, plus a Unix socket, enabling any language runtime — Go, Ruby, PHP, Elixir — to consume CDC events without a native SDK.
4. **Transform engine**: A QuickJS WASM runtime embeds a sandboxed JavaScript/TypeScript evaluation context inside `tap-core`, allowing user-defined transforms (filter, map, field-level mask) to run between capture and delivery.

Secondary additions include WebSocket and gRPC streaming sinks (complementing SSE), parallel snapshotting for large tables (chunked by primary key range), and three new CLI commands (`tap sidecar`, `tap deploy --target sidecar`, `tap transform`).

v0.2.0 also introduces two breaking changes inherited from its expanded architecture: (1) the `SourceMetadata.lsn` field becomes nullable (null for MySQL events — consumers that assumed non-null must update), and (2) `tap dev` defaults to sidecar mode unless `--embedded` is passed. The `ChangeEvent` envelope additionally gains a new `last_transform_evaluated_at` field, but this is **additive** (consumers that ignore unknown fields are unaffected) and is not counted among the breaking changes.

## 2. Scope — Boundaries

### 2.1 In Scope

| Deliverable | Rationale |
|---|---|
| **MySQL binlog adapter** (`tap-core`) | MySQL is the second-most-deployed relational database. ROW-based replication via binlog provides a reliable, low-overhead CDC mechanism comparable to Postgres logical replication. |
| **Parallel snapshotting** (`tap-core`) | Sequential snapshotting is a blocking bottleneck for large tables. Chunked parallelism by primary key range reduces initial snapshot time from hours to minutes at 10M-row scale. |
| **Transform engine: QuickJS WASM** (`tap-core`) | User-defined transforms are a foundational requirement for privacy (masking), routing, and enrichment. QuickJS is a compact, embeddable JS engine. Running it as WASM provides sandboxing and portability without an OS process boundary. |
| **Transform primitives: filter, map, mask** (`tap-core`) | The minimum set of transforms covering the most common use cases: dropping unwanted events, reshaping field values, and redacting sensitive fields before delivery. |
| **Python SDK via pyo3 + maturin** (`sdk-py`) | Python is the dominant language in data engineering and ML. `pyo3` provides zero-copy Rust-to-Python FFI; `maturin` handles wheel building across platforms. |
| **Asyncio-native `Tap` class** (`sdk-py`) | Modern Python async frameworks (FastAPI, Starlette, asyncio services) require an `async`/`await`-compatible API rather than a callback-only model. |
| **Pydantic model generation** (`sdk-py`) | Pydantic v2 is the de facto schema validation layer in Python. Auto-generating models from live database schema eliminates boilerplate and provides immediate type safety. |
| **Sidecar mode** (`tap-sidecar`) | Enables Go, Ruby, PHP, Elixir, and any HTTP-capable runtime to consume CDC events without a native SDK, by providing a local HTTP/SSE server on a well-known port. |
| **Unix socket IPC** (`tap-sidecar`) | Same-machine processes can connect via Unix socket for lower latency and without port conflicts, important in containerized environments and sidecars. |
| **Health + metrics endpoints** (`tap-sidecar`) | `GET /health` (readiness/liveness) and `GET /metrics` (Prometheus text format) are required for any container-based deployment workflow and integrate with Kubernetes liveness probes. |
| **WebSocket sink** (`tap-core`) | Browser applications and languages with better WebSocket support than SSE can subscribe to the event stream via standard WebSocket protocol. |
| **gRPC streaming sink** (`tap-core`) | gRPC provides strongly typed, bidirectional streaming and is the preferred transport for Go, Java, and Kotlin consumers. Defined in protobuf for cross-language compatibility. |
| **`tap sidecar` CLI command** (`tap-cli`) | Start the sidecar server from the CLI with full configuration control. |
| **`tap deploy --target sidecar` CLI command** (`tap-cli`) | Package the sidecar as a Docker container image ready for deployment, embedding config and binaries. |
| **`tap transform` CLI command** (`tap-cli`) | Validate transform scripts against a schema and test them against captured event fixtures offline. |
| **Full SDK binary matrix for Python** (`sdk-py`) | Prebuilt wheels for macOS ARM64, macOS x86_64, Linux x86_64 glibc, Linux ARM64 glibc, Linux x86_64 musl — via `maturin` matrix builds in CI. |

### 2.3 Known Limitations (v0.2.0)

| Limitation | Detail |
|---|---|
| **MySQL: ROW format required** | The MySQL binlog adapter requires `binlog_format=ROW` on the MySQL server. STATEMENT and MIXED formats are not supported. If `binlog_format` is not `ROW`, `tap init` will fail with an explicit error. |
| **MySQL: no GTID-based positioning** | MySQL capture binds to a single server's binlog coordinates `(binlog_file, binlog_offset)`. GTID-based positioning is out of scope for v0.2.0. As a consequence, capture does not survive MySQL failover or replica promotion: after a topology change the binlog coordinates are no longer valid and a re-snapshot is required. GTID support is deferred to a future release. |
| **MySQL: no `before` image on DELETE by default** | MySQL only provides the full before-image on DELETE if `binlog_row_image=FULL`. With `MINIMAL` or `NOBLOB`, the `before` field will be partially populated — only primary key columns are present in the before-image; all other columns are omitted. **Impact on mask transforms**: a mask transform targeting a `before.*` path (e.g., `before.email`) will fail-closed on every DELETE event from a `MINIMAL` primary (see §4.3.3). To use `before.*` mask fields, operators MUST set `binlog_row_image=FULL` (server-level setting, requires no restart but incurs a small binlog size increase — see §4.1.7 for the pre-flight check). |
| **Parallel snapshot: primary key required** | Chunked parallel snapshotting requires an integer or UUID primary key for range partitioning. Tables without such a key fall back to single-threaded sequential snapshot with a logged warning. |
| **Transform engine: stateless only** | The QuickJS WASM runtime executes one transform per event with no shared state across events. Stateful aggregations (windowing, joins) are not possible in v0.2.0 transforms. |
| **Sidecar: no TLS** | The sidecar binds to `localhost:9911` and the Unix socket path only by default (configurable via `sidecar.bind_address` — see §8.3). TLS is not supported in v0.2.0. Do not expose the sidecar port on a public interface unless you have explicitly bound to `0.0.0.0` and configured firewall rules. |
| **Python SDK: CPython only** | `pyo3` targets CPython 3.12+. PyPy, GraalPython, and other interpreters are not supported in v0.2.0. |
| **gRPC sink: server-streaming only (no client ACKs)** | The gRPC sink in v0.2.0 is server-streaming only. Client-streaming (consumer ACK flow control, backpressure) is deferred to v0.3.0. |
| **`wal2json` experimental** | The `wal2json` Postgres plugin remains experimental in v0.2.0. `pgoutput` is the recommended default. |

### 3.3 Transform Pipeline

Transforms are applied inside `tap-core` after the raw event is built from the source adapter and before the event is forwarded to any sink. The pipeline is synchronous within the event loop and operates on each event independently (stateless).

`[[transforms]]` is an **ordered list** executed in array order. Repeats are allowed — the same transform type may appear multiple times. Each transform runs only when the event's table and operation match the transform's `tables` and `ops` filters; events that do not match a transform's filters **skip that transform entirely** and continue to the next one (or to the sink if no transforms remain).

`last_transform_evaluated_at` is stamped at the epoch-ms timestamp of the **last transform that actually executed** (i.e., whose filters matched the event and whose function ran — including a `filter` that evaluated to `true` and passed the event through). The field is `null` in exactly two cases: (a) no transforms are configured for the stream, or (b) every configured transform was skipped due to table/op filter mismatch. In both cases the semantic is equivalent — no transform pipeline affected this event. When at least one transform matches and executes, the field is always stamped (regardless of whether the transform mutated the event). For a `filter` returning `true`, the timestamp reflects that the filter function ran; for a `filter` returning `false`, the event is dropped and the field is never serialized.

```
Source Adapter
     │
     │ raw ChangeEvent
     ▼
┌────────────────────────────────────────────────────┐
│               Transform Pipeline                   │
│                (ordered, repeatable)               │
│                                                    │
│  For each [[transforms]] entry (in array order):   │
│    ├── Check tables/ops filters                    │
```

| Criterion | Use `embedded` | Use `sidecar` |
|---|---|---|
| Single Python process consuming CDC | Preferred | Acceptable |
| Multiple services sharing one CDC stream | Not possible | Required |
| Non-Python services also consuming events | Not possible | Required |

| Separate deployment lifecycle for capture | Not possible | Required |

### 3.5 Data Flow Summary

**One-adapter-per-instance constraint**: Each `tap-core` process runs exactly one source adapter — either Postgres or MySQL, not both simultaneously. The `adapter` column in the SQLite `offsets` table records which adapter wrote each row; this exists for forward-compatibility with a shared state store across deployments, not to enable multi-source capture within a single process.

1. **Startup**: CLI or SDK initializes `tap-core`, loads the Tap config (search order per §9.5: `--config <PATH>` flag, then `.tap/config.toml`, then `./tap.toml`, then `TAP_CONFIG_PATH` env var), acquires a PID lock file at `.tap/state.db.lock`, then opens/creates SQLite state store at `.tap/state.db`. The PID lock prevents a second `tap-core` process from opening the same state store concurrently (see §8.4).
2. **Source selection**: Config selects Postgres (`adapter: "postgres"`) or MySQL (`adapter: "mysql"`). Only one may be active per process.
3. **Checkpoint recovery**: Reads last committed position (LSN for Postgres, binlog file+offset for MySQL) from SQLite `offsets` table.
4. **Snapshot phase**: If first run or incomplete snapshot, launches parallel snapshot workers (one per chunk) for tables with integer/UUID primary keys. Sequential fallback for other tables.
5. **Streaming phase**: Begins WAL/binlog streaming from the snapshot's commit position. Events decode into `ChangeEvent` structs.
6. **Transform phase**: Each `ChangeEvent` passes through the transform pipeline (filter → map → mask → stamp `last_transform_evaluated_at`). Filtered events are dropped before reaching the sink.
7. **Sink dispatch**: Surviving events are forwarded to all connected sinks (SSE, WebSocket, gRPC) concurrently.
8. **Checkpointing**: Every `flush_interval_ms` or `max_batch_size` events, the current confirmed position is flushed to SQLite.
9. **Shutdown**: SIGINT/SIGTERM → flush position → close source connections → drain sinks → exit.

## 4. Core Engine Changes

### 4.1 MySQL Binlog Adapter

**Location**: `crates/tap-core/src/adapters/mysql/`  
**New dependency**: `mysql_async` crate with the `binlog` feature enabled (resolved — see §14.1).

#### 4.1.1 Prerequisites and Server Configuration

The MySQL server must satisfy these conditions before `tap init` will succeed:

| Requirement | Configuration | Checked by `tap init`? |
|---|---|---|
| ROW-based binlog | `binlog_format=ROW` | Yes (hard error if not ROW) |
| Binlog retention | `expire_logs_days >= 7` (or `binlog_expire_logs_seconds`) | Yes (warning if < 7 days) |
| Full row image (recommended) | `binlog_row_image=FULL` | Yes (warning if not FULL) |
| MySQL user replication privilege | `GRANT REPLICATION SLAVE, REPLICATION CLIENT` | Yes (hard error if missing) |
| MySQL version | 5.7.x or 8.x | Yes (error if < 5.7) |
| MariaDB version | 10.5+ | Yes (error if < 10.5) |

#### 4.1.2 Binlog Event Processing

MySQL ROW-based replication emits events grouped in event sequences. The adapter processes the following event types:

| Binlog Event Type | Tap Operation | Notes |
|---|---|---|
| `WRITE_ROWS_EVENT` | `op: "c"` (insert) | Full after-image always available. |
| `UPDATE_ROWS_EVENT` | `op: "u"` (update) | Before and after images in one event (if `binlog_row_image=FULL`). |
| `DELETE_ROWS_EVENT` | `op: "d"` (delete) | Before-image available (if `binlog_row_image=FULL`); after is null. |
| `TABLE_MAP_EVENT` | Internal | Maps table ID to schema+table name. Cached per connection. |
| `ROTATE_EVENT` | Internal | Signals binlog file rotation. Updates checkpoint to new file+offset. |
| `XID_EVENT` | Transaction commit | Triggers LSN checkpoint flush. |
| `QUERY_EVENT` (DDL) | Logged only | DDL statements logged at WARN level. Schema cache invalidated. DDL capture is not first-class in v0.2.0. |

#### 4.1.3 Position Tracking

MySQL binlog position is a two-part identifier: `(binlog_file, binlog_offset)`. This is stored in the SQLite `offsets` table as a single serialized string in the `position` column: `"mysql-bin.000123:4567890"`. The v0.2.0 migration renames the existing `committed_lsn` column to `position` (see §8.4 and §10.2) so that both Postgres LSN strings and MySQL binlog coordinate strings share the same column.

**GTID positioning is out of scope for v0.2.0.** MySQL capture uses single-server `(binlog_file, binlog_offset)` coordinates only. These coordinates are not portable across topology changes: if the MySQL primary fails over to a replica, the binlog coordinates from the old primary are invalid on the new primary, and a full re-snapshot is required. GTID-based positioning, which would survive such a failover, is deferred to a future release (see §2.3).

The `SourceMetadata` struct gains a `binlog_file` field for MySQL events (populated when the source adapter is MySQL; null for Postgres events):

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SourceMetadata {
    pub db: String,
    pub schema: String,
    pub table: String,

    // Postgres: WAL LSN (e.g., "0/16B37428")
    // MySQL: null
    #[serde(skip_serializing_if = "Option::is_none")]
    pub lsn: Option<String>,

    // MySQL: binlog file name (e.g., "mysql-bin.000123")
    // Postgres: null
    #[serde(skip_serializing_if = "Option::is_none")]
    pub binlog_file: Option<String>,

    // MySQL: binlog byte offset
    // Postgres: null
    #[serde(skip_serializing_if = "Option::is_none")]
    pub binlog_offset: Option<u64>,

    pub tx_id: String,
    pub ts_ms: u64,

    #[serde(skip_serializing_if = "Option::is_none")]
    pub snapshot: Option<bool>,
}
```

#### 4.1.4 Schema Resolution

MySQL does not emit relation descriptor events the way Postgres's `pgoutput` does. The adapter resolves column names and types by querying `information_schema.columns` at connection time for each tracked table, then caching the result. On `QUERY_EVENT` (DDL detected), the schema cache for the affected table is invalidated and re-queried on the next relevant row event.

**Row size limit**: Events whose serialized `after` (or `before` for deletes) JSON payload exceeds `max_row_size_bytes` (default: 16 MB, configurable in `tap.toml`) emit a WARN-level log and are dropped from the stream with a counter increment (`tap_events_dropped_total{reason="row_too_large"}`) instead of being forwarded. This prevents a single oversized row from exhausting memory in the broadcast channel or sink buffers. The limit operates on the wire-encoded JSON byte length. Operators who expect larger rows (e.g., tables with `LONGBLOB` or `TEXT` columns storing multi-megabyte values) should increase this limit; values above 256 MB are not recommended as they increase peak memory pressure.

**Postgres json/jsonb encoding**: Postgres `json` and `jsonb` column values are serialized as native JSON values (object, array, string, number, boolean, or null) on the wire — not re-encoded as a JSON string. This matches MySQL's `JSON` encoding in §4.1.4a. Consumers receive a pre-parsed value; the SDK type is `unknown` (TypeScript) / `Any` (Pydantic). The adapter reads these from the DecodedLogicalReplicationMessage's binary jsonb representation, converting to a `serde_json::Value` at decode time.

#### 4.1.4b Big-Integer and High-Precision Wire Encoding

Two MySQL types — `BIGINT UNSIGNED` and `DECIMAL`/`NUMERIC` — as well as the Postgres `numeric` type exceed the range representable exactly by IEEE-754 double-precision floats (safe integer range: `[-(2^53-1), 2^53-1]`). Encoding these as JSON `number` would silently corrupt values at the boundary.

**Wire encoding rule**: `BIGINT UNSIGNED`, `DECIMAL`/`NUMERIC`, and Postgres `numeric` values are always serialized as **JSON strings** on the wire, regardless of their actual magnitude. This is a conservative choice that prevents loss for any value, not just edge cases.

**Consumer behavior**:
- TypeScript consumers receive these as `string`. Parse with `BigInt()` for exact integer arithmetic or a decimal library for `DECIMAL`.
- Pydantic consumers receive these as `str` and the generated model field type is `Decimal` (for `DECIMAL`/`numeric`) or `int` (for `BIGINT UNSIGNED`, with a `@field_validator`-decorated setter that accepts both `int` and `str` and converts to `int`). Python `int` has arbitrary precision (no IEEE-754 overflow), so the wire string is losslessly converted. The validator raises `ValueError` if the string is not a valid integer literal.
- gRPC `before_json`/`after_json` payloads carry the same string representation. Consumers must not deserialize these fields as `float64`.

#### 4.1.5 Idempotent Event IDs (MySQL)

| Event type | ID format |
|---|---|
| Streaming insert/update/delete | `"mysql:{binlog_file}:{binlog_offset}:{tx_id}"` |
| Snapshot row (single-col PK) | `"snap:{schema}.{table}:{pk_value}"` (same format as Postgres) |
| Snapshot row (composite PK) | `"snap:{schema}.{table}:{col1_val}:{col2_val}:..."` — columns in primary-key order, each value hex-encoded (lowercase hex of the wire-representation bytes). Hex encoding avoids ambiguity with the `:` delimiter. |

#### 4.1.6 Error Handling (MySQL-specific)

| Scenario | Behavior |
|---|---|
| `binlog_format` not ROW | Fatal error on `tap init`. Capture does not start. |
| Binlog position no longer available (retention expired) | Fatal error on startup. Log the missing position; user must re-run full snapshot. |
| Connection loss to MySQL | Same exponential backoff as Postgres (1s → 2s → ... → max 60s, 10 retries). |
| DDL encountered mid-stream | Schema cache invalidated. A synthetic `schema_change` event (op: `"m"`, see §10.1) is emitted to all connected sinks. Capture continues. |
| Table without primary key | Warning logged. Parallel snapshotting not available; sequential fallback used. Streaming capture continues normally. |

#### 4.1.7 Pre-flight Validation

On `tap dev` / `tap start`, the following pre-flight checks run and emit warnings or errors:

| Check | Condition | Action |
|---|---|---|
| MySQL `binlog_row_image` + mask `before.*` | If MySQL adapter is configured with `binlog_row_image != FULL` AND any mask transform references `before.*` fields, a FATAL error is emitted and the process exits. | The error message points the operator to §2.3: set `binlog_row_image=FULL` to use `before.*` mask transforms. The MySQL server variable is online-settable (`SET GLOBAL binlog_row_image = FULL`; no restart required). |
| Mask field path validity | For each mask `fields` dot-path targeting a specific column (e.g., `after.email`), the first path segment is validated against `information_schema.columns` for the referenced table. If the column does not exist, a `WARN` is emitted with the full path and the table name. | The capture continues; the mask will fail-closed on the first event where the path does not resolve. Operators should correct the path or add the column. |
| MySQL binary logging enabled | If MySQL adapter is configured and `SHOW MASTER STATUS` returns an empty result set (binary logging disabled), a `FATAL` error is emitted and the process exits. | Binary logging must be enabled (`log_bin=ON`) for MySQL CDC. If disabled, re-enable with `log_bin=ON` in `my.cnf` (requires restart) and re-create the replication user if needed. |
| MySQL `binlog_checksum` | If `@@binlog_checksum` returns `'CRC32'` (default since MySQL 5.6) AND the tap binlog parser does not support checksum verification, a `WARN` is emitted. | MySQL binlog events include a CRC32 checksum by default. The parser MUST strip (or verify) the 4-byte checksum footer from each binlog event. If the parser is not checksum-aware, binlog position tracking will diverge. The error message instructs operators to set `binlog_checksum=NONE` at session level (`SET @master_binlog_checksum= @@global.binlog_checksum`) until checksum-aware parsing is implemented. This is a WARN-level check because the default is safe for most connectors — the adapter should handle checksums transparently. |

Most pre-flight checks are advisory (WARN-level) so that partial configurations can start. MySQL binary-logging checks and other capability-gating checks are FATAL — if the source database cannot support CDC at all, the process exits immediately. Operators should treat WARN-level results as errors in production.

**Implementation note**: The `binlog_row_image` value is obtained by querying the MySQL server variable via `SHOW VARIABLES LIKE 'binlog_row_image'` (or equivalently `SELECT @@binlog_row_image`). This is a server variable, not a column query, and requires the MySQL user to have the `REPLICATION CLIENT` privilege (or `SUPER`). The `binlog_checksum` value is similarly obtained via `SELECT @@binlog_checksum` and is also a server variable requiring the same privilege level.

### 4.2 Parallel Snapshotting

#### 4.2.1 Algorithm

Parallel snapshotting replaces the single-threaded sequential approach for tables that have an integer or UUID primary key. The algorithm:

1. **Pre-compute chunk boundaries**: Execute `SELECT MIN(pk), MAX(pk) FROM {table}` to establish the full primary key range. Divide the range into `N` equal-width chunks where `N = min(num_cpus, max_snapshot_workers)` (default `max_snapshot_workers = 4`).
2. **Spawn worker tasks**: Each chunk range `[chunk_start, chunk_end)` is assigned to one Tokio task. Workers run concurrently.
3. **Worker query**: `SELECT * FROM {table} WHERE pk >= chunk_start AND pk < chunk_end ORDER BY pk`. Uses a snapshot-isolated transaction.
4. **Postgres consistency**: For Postgres, all worker connections share the same exported snapshot ID (`pg_export_snapshot()`) obtained on the coordinator before any worker starts. This guarantees all workers read the same consistent point-in-time view.
5. **MySQL consistency**: The coordinator takes `FLUSH TABLES WITH READ LOCK` first; every worker then issues `START TRANSACTION WITH CONSISTENT SNAPSHOT` while the lock is still held. Only after **all** workers have opened their consistent-snapshot transactions does the coordinator call `UNLOCK TABLES`. `SHOW MASTER STATUS` is recorded under lock to capture the binlog position for streaming hand-off. This protocol guarantees all workers read the same consistent point-in-time regardless of when each worker's `SELECT` physically executes.
6. **Progress tracking**: Each worker records its last emitted primary key value in the `snapshot_chunks` table (see Section 10.2) every `checkpoint_rows` rows (default 10,000). On resume, completed chunks are skipped and partial chunks resume from the saved offset.
7. **Completion**: When all worker tasks complete, the coordinator marks the table as fully snapshot in the `snapshots` table and begins streaming from the pre-recorded commit position.

#### 4.2.2 Chunk Strategy

```
Table: orders (50M rows, pk: id INTEGER, range: 1 to 50,000,000)
max_snapshot_workers: 4

Chunk 0:  id >= 1          AND id < 12,500,001
Chunk 1:  id >= 12,500,001 AND id < 25,000,001
Chunk 2:  id >= 25,000,001 AND id < 37,500,001
Chunk 3:  id >= 37,500,001 AND id < 50,000,001
```

For UUID primary keys, chunking uses range partitioning on the native column type: `uuid` for Postgres (which has a well-defined ordering), or `BINARY(16)` for MySQL (stored as raw bytes). The MIN/MAX range query operates on the native type; chunk boundaries are stored as their native binary or UUID representation, not as lexicographic strings. Skewed UUID distributions (e.g., time-ordered UUIDs such as UUIDv7 concentrated in one range) will yield severely imbalanced chunks — one worker may process >90% of rows while others idle. For UUIDv7 primary keys (the most common skewed case), `tap init` MUST pre-flight check the distribution by sampling 10k rows and emit a `WARN` if stddev of chunk sizes exceeds 2x the mean. Operators seeing this warning should set `snapshot_mode = "sequential"` or increase `max_snapshot_workers` only if the source data is truly uniform. The §11.1 throughput targets assume uniform distribution; skewed distributions will fall short of the 1.5x target by an order of magnitude.

For tables with a non-integer, non-UUID primary key, or no primary key, the adapter logs a warning and falls back to single-threaded sequential snapshot.

#### 4.2.3 LSN/Binlog Consistency Guarantees

The parallel snapshot must guarantee that the streaming phase starts at the exact commit position captured at snapshot initiation, with no events missed and no events duplicated between snapshot and streaming phases.

**Postgres guarantee**:
- `pg_export_snapshot()` is called once by the coordinator before any worker starts.
- All workers use `SET TRANSACTION SNAPSHOT '{snapshot_id}'`.
- The coordinator records the LSN at snapshot export time.
- Streaming begins at that LSN.
- Any WAL records after that LSN but committed before streaming begins are captured by the streaming phase; none are missed.

**MySQL guarantee**:
- Coordinator acquires `FLUSH TABLES WITH READ LOCK` on its connection.
- `SHOW MASTER STATUS` records the current binlog position (this is the streaming hand-off point).
- Every worker issues `START TRANSACTION WITH CONSISTENT SNAPSHOT` while the lock is still held. InnoDB assigns each worker the same consistent read view because the global lock prevents any commit from advancing the MVCC horizon during this window.
- Only after all workers have opened their consistent-snapshot transactions does the coordinator call `UNLOCK TABLES`.
- Workers then execute their `SELECT` queries. The lock is no longer held during the actual data scan.
- Streaming begins at the recorded binlog position.
- **Residual risk**: The window during which the global write lock is held is bounded by worker startup time (connection setup + `START TRANSACTION`), not by the duration of the full table scan. For `max_snapshot_workers = 4` on a network-attached MySQL primary with sub-millisecond RTT, this window is typically 50-200ms; on a high-latency link or with `max_snapshot_workers = 16` it may exceed 5 seconds. The actual lock duration is exposed as `tap_snapshot_lock_held_ms` (see §5.2.3); operators should alert if this exceeds their write-latency SLO. Operators with strict write-latency SLAs should reduce `max_snapshot_workers` or schedule snapshots during low-traffic periods.
- **Worker-startup timeout**: If a snapshot worker fails to open its consistent-snapshot transaction within `snapshot_worker_startup_timeout` (default: 30 s), the coordinator aborts the snapshot: it calls `UNLOCK TABLES`, logs an error with the worker index and elapsed time, and exits with a non-zero status. This prevents an unresponsive worker from holding the global write lock indefinitely.

**Postgres vs. MySQL asymmetry**: Postgres uses `pg_export_snapshot()` combined with `SET TRANSACTION SNAPSHOT` to share a consistent snapshot across workers without any global lock. MySQL requires the lock-until-all-workers-open protocol above because InnoDB's consistent read view is established at transaction start time, and there is no equivalent snapshot-export mechanism. The MySQL approach holds a write-blocking lock only during worker startup; Postgres holds no write-blocking lock at all.

#### 4.2.4 New Configuration Parameters

| Parameter | Default | Description |
|---|---|---|
| `max_snapshot_workers` | `4` | Maximum concurrent snapshot worker tasks per table |
| `chunk_size_rows` | `0` (auto) | Override chunk size in rows. `0` = auto-divide range by worker count. |
| `checkpoint_rows` | `10000` | Rows emitted per worker before saving progress to SQLite |
| `snapshot_mode` | `"parallel"` | `"parallel"` (default) or `"sequential"` (v0.1.0 behavior) |
| `snapshot_worker_startup_timeout` | `30` | Seconds to wait for each snapshot worker to open its consistent-snapshot transaction before aborting the snapshot. Only applies to MySQL (FLUSH TABLES WITH READ LOCK protocol); Postgres does not hold a global lock. |

#### 4.2.5 Transforms and Snapshot Events

By default, `filter` and `map` transforms are **skipped for snapshot events** (`op: "r"`). Snapshot events exist solely to populate the initial state; applying `filter` or `map` transforms to them can discard rows or mutate data in ways that corrupt the consumer's initial view. The `skip_transforms_on_snapshot` option defaults to `true`.

**Exception — `mask` always runs on snapshot events**: Mask transforms are privacy-critical and apply unconditionally, regardless of `skip_transforms_on_snapshot`. A `mask` on `after.email` redacts the email field for every snapshot row streamed to any sink. This carve-out exists because bypassing privacy transforms on the initial snapshot would leak unredacted PII to all connected consumers.

To apply `filter` and `map` transforms to snapshot events, set `skip_transforms_on_snapshot = false` in `tap.toml`. When doing so, ensure `filter` transforms do not discard snapshot rows unintentionally. The §11.1 throughput targets and §11.2 latency targets are measured with the default `skip_transforms_on_snapshot = true`; enabling `filter`/`map` on snapshot events will reduce snapshot throughput proportionally to transform overhead.

### 4.3 Transform Engine

#### 4.3.1 QuickJS WASM Runtime

**Location**: `crates/tap-core/src/transforms/`  
**Engine**: QuickJS compiled to WASM, embedded as a static binary blob. The WASM module is loaded once at `tap-core` initialization. Transform scripts are compiled to QuickJS bytecode **once at load time** (not per event). Per-event execution either resets globals on a pre-warmed context or creates a lightweight realm over the precompiled bytecode, depending on the embedding strategy chosen during implementation. Either way, bytecode compilation is not repeated per event. This is the key performance difference from a naive "new context per event" approach.

**Timeout enforcement**: The 100 ms per-event budget (configurable via `transform_timeout_ms`) is enforced via the QuickJS interrupt handler callback (if using the native QuickJS C API) or via wasmtime epoch-based interruption (if embedding QuickJS as a WASM module inside wasmtime). The interrupt handler fires at regular instruction-count intervals and checks elapsed wall time; if the budget is exceeded, execution is interrupted and the event is passed through unmodified (fail-open) with a warning logged. This mechanism is checked during implementation against QuickJS's C API and wasmtime's epoch API — see §14.2.

**Rationale for QuickJS over V8 / SpiderMonkey**:

| Criterion | QuickJS | V8 |
|---|---|---|
| Binary size (embedded) | ~660 KB WASM | ~30 MB |
| Startup time | < 1 ms (cold) | ~10–50 ms (cold) |
| Sandboxing | WASM sandbox (OS-enforced) | Requires V8 isolates (higher complexity) |
| ES2020 compliance | Yes | Yes |
| TypeScript (transpiled) | Yes (pre-transpile at `tap transform validate`) | Yes |
| Async transforms | No (v0.2.0 — stateless sync only) | Yes |

TypeScript transforms must be transpiled to ES2020-compatible JavaScript before runtime execution. `tap transform validate` performs this transpilation using the `swc_core` crate (already used by the TypeScript SDK toolchain).

#### 4.3.2 Transform Descriptor Schema

Transform configurations are declared in `tap.toml` under the `[[transforms]]` array:

```toml
[[transforms]]

[[transforms]]
```


#### 4.3.3 Transform Primitives

**`filter`**: Evaluates a predicate. If the function returns `false` (or falsy), the event is dropped and does not reach any sink. The `last_transform_evaluated_at` field is NOT set on dropped events (they never emit).

```javascript
// Example: drop snapshot events for the audit_log table
event => !(event.source.table === 'audit_log' && event.source.snapshot === true)
```

**`map`**: Evaluates a mapping function. Returns a new event object. The function may add fields, rename fields, or compute derived values. It must return an object with at least `op`, `source`, and `ts_ms` fields intact.

```javascript
// Example: add a computed field and drop a sensitive column
event => ({
  ...event,
  after: event.after ? { ...event.after, full_name: `${event.after.first_name} ${event.after.last_name}`, ssn: undefined } : null
})
```

**`mask`**: Replaces the value at each specified dot-path field with the strategy output. Does not require a script file — operates on configuration alone.

| Strategy | Behavior |
|---|---|
| `"redact"` | Replaces field value with the string `"***REDACTED***"` |
| `"hash"` | Replaces field value with `HMAC-SHA-256(stable_salt, to_string(value))` hex-encoded. The stable salt is read from `tap.toml` key `mask_salt` (a required non-empty string; `tap init` generates a random 32-byte hex value). The salt is **not** ephemeral — it must be identical across restarts so that hashed values remain consistent for deduplication. The salt is stored in `tap.toml`, not in the SQLite state store. Operators must back up `tap.toml` to preserve hash consistency. See also §4.1.7 — MySQL partial before-images may mean `before.*` mask dot-paths resolve to absent fields when `binlog_row_image` is not `FULL`. |
| `"null"` | Replaces field value with JSON `null` |

**Absent dot-path behavior**: If a mask `fields` dot-path does not resolve to a present field in the event (e.g., `"after.email"` on a DELETE event where `after` is null, or `"before.ssn"` on a MySQL INSERT where the before-image is absent), the mask **fails closed**: the event is rejected with a warning log entry and is not forwarded to any sink. To suppress this for known-absent paths, set `mask_missing_field = "skip"` in the transform descriptor (default: `"error"`). Cross-reference: MySQL `binlog_row_image=MINIMAL` or `NOBLOB` may cause `before.*` paths to be absent on DELETE events. With `binlog_row_image=MINIMAL`, only the primary key columns of the row are written to the before-image; all other columns are omitted. This is a footgun: a mask transform on `before.email` will silently fail-closed (reject every DELETE event) on a `MINIMAL` MySQL primary. To use `before.*` mask fields, operators MUST set `binlog_row_image=FULL` (see §2.3 for the operational impact — this is a server-level setting that requires no MySQL restart but does incur a small binlog size increase). The pre-flight check in §4.1.7 emits a `WARN` at config-load time if this combination is detected.

#### 4.3.4 Post-Transform Envelope Validation

After the full transform pipeline completes (all `[[transforms]]` entries have run), the resulting event is validated against the minimum required envelope before any sink receives it. A `map` transform that drops or renames required top-level fields will cause the event to be rejected at this stage.

Required fields that must survive the transform pipeline:

| Field | Rejection behavior if absent or wrong type |
|---|---|
| `op` | Event rejected; logged at ERROR with transform path and event ID |
| `source` | Event rejected; logged at ERROR |
| `ts_ms` | Event rejected; logged at ERROR |

Rejected events are not forwarded to any sink. The rejection is counted in `tap_events_filtered_total`. This validation runs even when `transform_on_error = "pass"` is configured, because a structurally invalid envelope is a data correctness issue, not a transform execution error.

#### 4.3.5 Sandboxing and Security

- Each QuickJS WASM context has no access to the host filesystem, network, or environment variables. The only input is the serialized `ChangeEvent` JSON and the only output is a modified JSON value (or null for filter=false).
- Transform execution time limit: 100 ms per event. If a transform exceeds this limit, the event is passed through unmodified and a warning is logged with the transform path and event ID. This limit is configurable via `transform_timeout_ms` in `tap.toml` (default: `100`). Enforcement mechanism: QuickJS interrupt handler or wasmtime epoch interruption — see §4.3.1.
- Memory limit per QuickJS context: 16 MB (configurable via `transform_memory_limit_mb`, default: `16`). If the memory limit is exceeded, the transform is treated equivalently to a thrown exception — the event is passed through or dropped according to `transform_on_error` (fail-open by default for `filter`/`map`, fail-closed for `mask`). This is consistent with the `transform_on_error` semantics rather than a separate fail mode.
- **`filter`, `map` (fail-open by default)**: Transforms that throw exceptions log the error with event ID and pass the event through unmodified. This is the safe default for non-privacy transforms: dropping an event on a JavaScript error is worse than emitting a row the consumer may not have wanted. Fail-closed behavior (drop the event on transform error) can be configured via `transform_on_error = "drop"` in `tap.toml`.
- **`mask` (fail-closed by default)**: Masking transforms that throw exceptions, encounter an unreadable field, or fail to compute a hash REJECT the event (do not emit it to any sink). The reasoning: a `mask` transform is privacy-critical — emitting an event with unmasked PII because the transform crashed is a worse failure mode than dropping a single event. The rejected event is logged at `WARN` with event ID, the failing transform path, and the exception. The default is non-configurable. Per-transform override is available via the `mask_on_error` field in the transform descriptor: `"drop"` (default, fail-closed) or `"pass"` (fail-open, log only). This split is independent of `transform_on_error` (which controls `filter`/`map`).

**§11.2 latency re-baseline**: The overhead targets in §11.2 assume compile-once-at-load bytecode. A naive "new context per event" approach would add ~1–5 ms cold-compilation overhead per transform call, making the `< 1 ms` `map` target unreachable. The compile-once model is a prerequisite for the §11.2 targets.

## 4.4 Replication Slot Lifecycle

**Location**: `crates/tap-core/src/postgres/connection.rs`  
**Related**: [`tap_replication_slot_wal_retained_bytes`](#table-523-prometheus-metrics) in §5.2.3

### 4.4.1 Design Intent

Tap preserves the Postgres logical replication slot across restarts to enable **resume capability**: on restart, Tap reads the last committed position from the SQLite state store and resumes the WAL stream from that LSN. Dropping the slot on shutdown would force a full re-snapshot on every restart.

However, an inactive replication slot prevents Postgres from recycling Write-Ahead Log (WAL) files. Every WAL segment from the slot's `restart_lsn` to the current write position must be retained, because the slot promises the subscriber will eventually consume those changes. Over time (hours to days, depending on write volume), accumulated WAL can exhaust available disk space — even when the actual database size is small (~18 MB data grew 1,056 MB WAL in a verified incident on Aiven).

### 4.4.2 Slot Lifecycle Summary

| Phase | Slot State | Default Behavior |
|-------|-----------|------------------|
| `tap init` / first start | `CREATE_REPLICATION_SLOT` | Created with `pgoutput` plugin |
| `tap capture` running | `active = true` | WAL consumed, slot lag near zero |
| `tap capture` stopped gracefully | `active = false` | Slot preserved for resume; WAL accumulates |
| External slot drop | Removed from `pg_replication_slots` | Next start re-creates slot (fresh start) |
| Database dropped | Orphaned | Must be manually cleaned up |

### 4.4.3 WAL Retention Risk

The risk is proportional to write volume × slot inactivity duration:

```
wal_disk_usage ≈ write_throughput × seconds_since_last_activity
```

On a database with moderate write throughput (e.g., 100 KB/s), a slot left inactive for 24 hours retains ~8.4 GB of WAL. The Aiven incident reached 100% disk usage (1,056 MB WAL on ~18 MB data) in less than 48 hours.

**PostgreSQL has no built-in WAL retention limit per slot.** The slot holds WAL indefinitely until the consumer advances `confirmed_flush_lsn` or the slot is dropped.

### 4.4.4 Mitigation in v0.2.0

| Mitigation | How |
|------------|-----|
| **Monitoring** (primary) | `tap_replication_slot_wal_retained_bytes` metric in `/metrics` (§5.2.3). Alert on values above threshold. |
| **Manual cleanup** | SQL: `SELECT pg_drop_replication_slot('tap_slot');` |
| **Documentation** | See §9.6 for lifecycle docs and cleanup procedures. |

**v0.2.0 does not auto-drop the slot on shutdown** (intentional — resume requires it). The monitoring metric is the recommended defense.

### 4.4.5 MySQL Note

MySQL does not have a replication slot concept. Binlog retention is controlled by `expire_logs_days` / `binlog_expire_logs_seconds` on the MySQL server, which is a global retention policy, not a per-connector marker. The binlog adapter uses `(binlog_file, binlog_offset)` coordinates and does not hold a persistent marker on the server. The WAL-retention risk described in this section applies to **Postgres only**.

## 5. Sidecar Mode

### 5.1 `tap-sidecar` Binary

**Location**: `crates/tap-sidecar/`  
**Key dependencies**: `axum` (HTTP server), `tokio` (async runtime), `tap-core` (library), `prometheus` (metrics), `tokio-tungstenite` (WebSocket).

The sidecar binary is a thin HTTP/WebSocket/Unix-socket wrapper around `tap-core`. It starts a single instance of the capture engine internally and multiplexes events to all connected clients.

### 5.2 HTTP API Surface

All endpoints bind to `localhost:9911` by default. The port is configurable via `--port` or `sidecar.port` in `tap.toml`.

#### 5.2.1 Control Endpoints

| Method | Path | Description |
|---|---|---|
| `POST` | `/capture/start` | Start capture (if not already running). Body: optional JSON config override. Returns `202 Accepted`. |
| `POST` | `/capture/stop` | Stop capture gracefully. Flushes checkpoint. Returns `202 Accepted`. |
| `POST` | `/capture/pause` | Pause WAL/binlog reading. Backpressure signal. Returns `200 OK`. |
| `POST` | `/capture/resume` | Resume paused capture. Returns `200 OK`. |

**Authentication**: None in v0.2.0 (see §7.1). Control endpoints bind to loopback by default (`127.0.0.1`), which provides network isolation but does not prevent local access: any process or user on the same machine can stop or pause capture. This is acceptable for single-user dev environments. Production deployments MUST set `sidecar.bind_address` per §8.3 and front the HTTP port with a reverse proxy that authenticates control calls, or use a Unix socket with filesystem permissions (§5.3).

#### 5.2.2 Event Stream Endpoints

| Method | Path | Protocol | Description |
|---|---|---|---|
| `GET` | `/events` | HTTP/1.1 or HTTP/2, `text/event-stream` | SSE stream. Each event is a JSON-encoded `ChangeEvent`. |
| `GET` | `/ws/events` | WebSocket (RFC 6455 upgrade) | WebSocket stream. Each message is a JSON-encoded `ChangeEvent`. |

**SSE stream format**:

```
event: change
data: {"op":"c","after":{"id":1,"name":"Alice"},"source":{...},"ts_ms":1717200000150,"id":"0/16B37428:12345","last_transform_evaluated_at":null}
id: 0/16B37428:12345

event: change
data: {"op":"u",...}
id: 0/16B37430:12346
```

Query parameters for `/events` and `/ws/events`:

| Parameter | Type | Description |
|---|---|---|
| `tables` | `string` | Comma-separated table filter: `?tables=public.users,public.orders` |
| `ops` | `string` | Operation filter: `?ops=c,u` (omit deletes) |
| `last_event_id` | `string` | Resume from a specific event ID (reconnect support for SSE) |

**`last_event_id` eviction contract**: The sidecar holds a bounded in-memory event buffer (`max_buffer_size`, default 10,000 events). When a client reconnects with a `last_event_id` that is older than the oldest event in the buffer (i.e., the event has been evicted), the sidecar **emits an explicit SSE `error` event** before beginning the stream:

```
event: error
data: {"code":"event_id_evicted","last_event_id":"0/16B00000:11000","message":"Requested event ID is no longer in the server buffer. Stream will resume from head. Gap-free replay requires v0.3.0 DLQ."}
```

After emitting the error event, the server begins streaming from the current head of the buffer (no silent jump). The client can inspect the error event and decide whether to abort or accept the gap. **Gap-free resume is not guaranteed in v0.2.0**; it requires the DLQ feature deferred to v0.3.0.

**Counter reset-on-restart semantics**: `tap_events_total`, `tap_events_filtered_total`, and all other Prometheus counters reset to zero on each sidecar restart. Prometheus and compatible systems detect counter resets via monotonicity breaks; downstream recording rules that compute rates (`rate()`, `increase()`) handle resets correctly by design. Alert on `rate()` values, not on raw counter drops.

**Label cardinality**: `tap_snapshot_progress_pct{table=...}` emits one time series per tracked table, bounded by the number of tables in the capture filter. Do not add high-cardinality values (row keys, event IDs) as label values. The `table` label value is the fully qualified table name (e.g., `"public.orders"`).

#### 5.2.4 Structured Logging

The sidecar emits logs in **newline-delimited JSON** (ndjson) format, one line per log event:

```json
{"ts":"2026-06-06T12:00:01.234Z","level":"info","msg":"capture started","correlation_id":"tap_capture_001","adapter":"postgres","db":"mydb"}
{"ts":"2026-06-06T12:00:02.567Z","level":"warn","msg":"transform_error","correlation_id":"tap_capture_001","event_id":"0/16B37430:10001","transform":"redact_email","elapsed_ms":105}
{"ts":"2026-06-06T12:00:03.890Z","level":"error","msg":"connection lost; retrying","correlation_id":"tap_capture_001","retry":3,"max_retries":10}
```

**Fields**:
- `ts`: ISO 8601 UTC timestamp with millisecond precision.
- `level`: One of `trace`, `debug`, `info`, `warn`, `error`.
- `msg`: Short, static event name (e.g., `"capture started"`, `"transform_error"`, `"snapshot_chunk_done"`).
- `correlation_id`: Scoped to a single capture session; stable across restarts when read from state.db or provided via `--correlation-id`. Enables log correlation across sidecar restarts in CI/CD pipelines.

**Per-event context fields**: Log lines that reference a specific change event include an `event_id` field matching the event's `ChangeEvent.id` (§10.1). Transform errors, row-size drops, and mask failures all include `event_id` for traceability.

**Log level is configurable** via `--log-level` (default: `info`) or `sidecar.log_level` in config. At `debug` level, each emitted event produces a log line with its `event_id`, `op`, and `table` — useful for development but too verbose for production.

#### 5.2.6 Backfill and Replay

The SSE `/events` endpoint supports event replay via the `last_event_id` query parameter (see §5.2.2). This replay is **buffer-bound**: the sidecar can only replay events that are still present in its in-memory ring buffer (`max_buffer_size`, default 10,000 events). Events evicted from the buffer cannot be replayed — the server returns an explicit error event and the client must accept the gap.

**Durable backfill (re-emitting historical snapshot data) is not available as a standalone API in v0.2.0.** The only mechanism to obtain historical data is to trigger a full re-snapshot:

- Delete the `.tap/state.db` state store and restart capture (see §8.4).
- Or set `snapshot_mode = "sequential"` in `tap.toml` and force a re-snapshot by clearing the `snapshots` table (`DELETE FROM snapshots; DELETE FROM snapshot_chunks;`). **Warning**: This does NOT reset `offsets.position` — the re-snapshot runs from the current database position, not from the earliest available history. To obtain a full backfill (all rows), also reset `offsets.position` to the earliest binlog position or truncate the `offsets` table entirely.

Both approaches consume the streaming phase from the current database position, so events emitted during the re-snapshot are interleaved with live CDC events. Consumers receiving snapshot rows (`op: "r"`) alongside streaming rows (`c`/`u`/`d`) must deduplicate by event ID. Data loss is possible if consumers do not account for the gap between the latest snapshot and the concurrent streaming phase.

**Planned for v0.3.0**: A dedicated re-snapshot REST endpoint (`POST /capture/resnapshot`), selective table re-snapshot, and a durable DLQ-backed replay buffer that survives sidecar restarts.

### 6.2 Asyncio Integration

The Python SDK runs the Tokio async runtime on a dedicated OS thread and bridges Rust futures to Python's asyncio event loop using a bespoke channel bridge (see §14.4 — resolved). Events arrive in Python as `asyncio.Queue` items, allowing native `async for` iteration.

```python
# Internal bridge (simplified)
class Tap:
    def __init__(self, config, mode="embedded", **kwargs):
        self._inner = _tap_cdc.TapCore(config, mode, **kwargs)
        # Bounded queue — default 10,000 events. When full, Rust sender blocks,
        # applying backpressure to the capture engine (mirrors v0.1.0 behavior).
        self._queue: asyncio.Queue = asyncio.Queue(maxsize=10_000)

    async def stream(self) -> AsyncIterator[ChangeEvent]:
        loop = asyncio.get_running_loop()  # NOT get_event_loop() — requires running loop
        await self._inner.start(loop=loop, queue=self._queue)
        try:
            while True:
                event = await self._queue.get()
                if event is None:  # sentinel: capture stopped
                    break
                yield ChangeEvent.model_validate(event)
        except asyncio.CancelledError:
            # CancelledError propagates normally; stop is called in finally.
            raise
        finally:
            # Waits for Tokio thread to join before returning. This ensures the
            # background thread does not outlive the Python coroutine.
            await self._inner.stop()
```

**Thread model**: The Tokio runtime runs on a dedicated background OS thread started at `Tap.__init__`. Rust-to-Python event delivery uses a bespoke bridge: the Rust sender writes serialized event JSON into an `mpsc` channel, and a watcher task calls `loop.call_soon_threadsafe(queue.put_nowait, event_json)` to schedule delivery onto the asyncio event loop. The GIL is held only briefly during the `call_soon_threadsafe` call (the Python queue put); it is released during Rust-side event processing and transform evaluation. This approach is preferred over `pyo3-asyncio` because it avoids `pyo3-asyncio`'s pinned `tokio` and `pyo3` version matrix, which has historically caused compatibility failures (see §14.4 — resolved).

**Backpressure**: The asyncio queue is bounded (`maxsize=10_000` by default, configurable via `queue_max_size`). When the queue is full, `call_soon_threadsafe` cannot add items and the Tokio channel's send side will block (or apply bounded-buffer backpressure depending on channel type). This pauses WAL/binlog reading in the Rust layer, mirroring the v0.1.0 behavior.

**Shutdown under `asyncio.CancelledError` / `KeyboardInterrupt`**: `asyncio.CancelledError` is re-raised after `stop()` completes. `KeyboardInterrupt` delivered to the main thread signals the Tokio runtime to shut down; `stop()` joins the Tokio thread before returning. In both cases, the final checkpoint is flushed before the background thread exits.

## 7. Transport Layer

### 7.1 WebSocket Sink

**Protocol**: RFC 6455 WebSocket, upgradable from HTTP/1.1 or HTTP/2.  
**Endpoint**: `/ws/events` (sidecar mode) or a configurable local port in embedded mode.  
**Message format**: Each WebSocket message is a UTF-8 JSON string containing one serialized `ChangeEvent`. Binary frames are not used.

**Connection lifecycle**:
1. Client connects via WebSocket handshake (HTTP Upgrade).
2. Server accepts and begins streaming events.
3. Server sends a `ping` frame every 30 seconds (configurable via `ws_ping_interval_ms`). Client is expected to respond with `pong`.
4. If the client does not respond to three consecutive pings, the server closes the connection with status 1001 (Going Away).
5. On server shutdown, the server sends a `close` frame with status 1001 before closing.

**Reconnection responsibility**: The client is responsible for reconnecting. The server does not buffer events for disconnected WebSocket clients beyond the in-memory event buffer (`max_buffer_size`, default 10,000 events). Events that overflow the buffer during a client disconnect are dropped from that client's perspective only; other connected clients are unaffected.

**Asymmetry with SSE**: Unlike the SSE stream (which supports `last_event_id` resume — see §5.2.2), WebSocket has no built-in resume mechanism. A WebSocket client that disconnects and reconnects receives the stream from the current buffer head; events evicted before the reconnect are lost from that client's perspective. Clients requiring gap-free replay should use SSE with `last_event_id` or the gRPC sink with `last_event_id` (see §7.2). Note that both SSE and gRPC rely on the same in-memory buffer (§7.3) — true gap-free replay across buffer evictions requires the v0.3.0 DLQ.

**Authentication**: None in v0.2.0. TBD in v0.6.0.

### 7.2 gRPC Streaming Sink

**Location**: `crates/tap-proto/` (protobuf definitions), `crates/tap-core/src/sinks/grpc.rs` (server implementation).  
**Port**: `9912` (dedicated port, separate from the HTTP/SSE/WebSocket sidecar port `9911`). Using a separate port avoids requiring `axum` to multiplex HTTP/1.1 (SSE) and HTTP/2-only (gRPC) traffic on the same socket.

**Honest framing**: The gRPC sink uses **binary HTTP/2 framing with a JSON string payload** — it is not strongly typed in the protobuf sense. Row data (`before_json`, `after_json`) is carried as serialized JSON strings inside the protobuf message, not as native protobuf fields. This trades schema safety for simplicity: schema-native encoding would require a schema registry to generate message types at runtime. Schema-native gRPC payloads are deferred to v0.3.0. The transport is **server-streaming only** (`rpc Subscribe(...) returns (stream ChangeEvent)`); client-streaming (consumer ACK flow control) is deferred to v0.3.0.

**Proto definition**:

```protobuf
syntax = "proto3";
package tap.v1;

option go_package = "github.com/tap-cdc/tap/gen/go/tap/v1;tapv1";
option java_package = "com.tapcdc.tap.v1";

service TapService {
  // Server-streaming RPC: client subscribes and receives a stream of events.
  rpc Subscribe(SubscribeRequest) returns (stream ChangeEvent);
}

message SubscribeRequest {
  // Optional: filter to specific tables (e.g., "public.users")
  repeated string tables = 1;
  // Optional: filter to specific operations ("c", "u", "d", "r")
  repeated string ops = 2;
  // Optional: resume from a specific event ID
  string last_event_id = 3;
}

message SourceMetadata {
  string db = 1;
  string schema = 2;
  string table = 3;
  optional string lsn = 4;
  optional string binlog_file = 5;
  optional uint64 binlog_offset = 6;
  string tx_id = 7;
  uint64 ts_ms = 8;
  optional bool snapshot = 9;
}

message ChangeEvent {
  string op = 1;
  // before and after are JSON-encoded strings (not nested proto messages)
  // to avoid re-defining every table's schema in protobuf.
  optional string before_json = 2;
  optional string after_json = 3;
  SourceMetadata source = 4;
  uint64 ts_ms = 5;
  string id = 6;
  optional uint64 last_transform_evaluated_at = 7;
}
```

**Rationale for JSON-encoded `before_json`/`after_json`**: Table schemas are not known at proto compilation time and vary per database. Encoding row data as JSON strings within the protobuf message avoids requiring a schema registry to dynamically generate message types. This trades some efficiency for simplicity. Schema-native encoding (using `google.protobuf.Struct` or dynamic protobuf) is a v0.3.0 consideration.

**`last_event_id` semantics**: When a `SubscribeRequest` includes `last_event_id`, the gRPC server searches the in-memory event buffer for the event matching that ID. If found, the stream resumes from the event immediately after it. If the event ID is no longer in the buffer (evicted), the server sends a gRPC error of type `OUT_OF_RANGE` with message `"requested event_id is no longer in the server buffer"` before closing the stream — mirroring the SSE error-event contract in §5.2.2. The client must then decide whether to re-subscribe without `last_event_id` (accepting the gap) or initiate a re-snapshot. Gap-free replay across buffer evictions requires the v0.3.0 DLQ.

**`max_message_size`**: tonic (the gRPC implementation) defaults to 4 MB per message. Large CDC events — primarily snapshot rows with many columns or wide `TEXT`/`BLOB` values — may exceed this limit. The sidecar sets `max_message_size` to 16 MB (matching the `max_row_size_bytes` default in §4.1.1) via tonic's `max_frame_size` / `max_decoding_message_size` configuration. Operators who raise `max_row_size_bytes` above 16 MB MUST also increase the gRPC message size limit via `sidecar.grpc_max_message_size_bytes` in `tap.toml`; values above 256 MB are not recommended.

**Proto3 field behavior — `snapshot` presence**: In the JSON envelope (§10.1), `snapshot` is **omitted** when the event is not from a snapshot (`skip_serializing_if`). In proto3, the `optional` keyword (v3.15+) enables explicit field presence: an unset `optional bool snapshot = 9` is **absent** from the wire format — it is NOT serialized as `false`. This means gRPC consumers see `snapshot` only on snapshot events (absent on non-snapshot events), matching JSON behavior. Both JSON and gRPC consumers MUST check `snapshot === true` (explicit truth) rather than checking for the key's presence, since JSON omits the key on non-snapshot events and proto3 omits the field on non-snapshot events. An explicit `false` value (set by the server) is distinguishable from absence but should be treated equivalently — only `true` indicates a snapshot event.

**gRPC server in embedded mode**: In embedded mode, the gRPC server runs on an ephemeral port chosen at startup. The port is logged at startup and available via the `TapCore` API:

```typescript
const port = await tap.grpcPort() // returns e.g. 50051
```

```python
port = await tap.grpc_port()
```

### 7.3 Coexistence of SSE, WebSocket, and gRPC

All three sinks share a single internal broadcast channel (`tokio::sync::broadcast`). The event is serialized once into a `ChangeEvent` struct; each sink serializes it independently into its own wire format. Transforms run once before the broadcast, so all sinks receive the same transformed event.

```
tap-core event loop
      │
      │ (single ChangeEvent, post-transform)
      ▼
broadcast::Sender<ChangeEvent>
      │
      ├──► SSE serializer → text/event-stream
      ├──► WebSocket serializer → JSON UTF-8 frame
      └──► gRPC serializer → protobuf ChangeEvent message
```

A client connected only via WebSocket does not block the SSE or gRPC sinks. Slow consumers on any sink apply backpressure to their own channel receiver only; the broadcast channel's lagging-receiver behavior is configured via `max_buffer_size`.

**Slow consumer overflow behavior**: When a sink's receiver lags behind the broadcast sender by more than `max_buffer_size` (default: 10,000 events), `tokio::sync::broadcast::Receiver::recv()` returns a `Lagged(n)` error indicating `n` events were dropped. Each sink handles this differently:

- **SSE sink**: Logs `"SSE client disconnected (buffer overflow)"` at WARN level, closes the connection, and increments `tap_events_dropped_total{reason="sink_buffer_overflow"}`. The client may reconnect with `last_event_id` for gap recovery.
- **WebSocket sink**: Logs `"WebSocket client disconnected (buffer overflow)"` at WARN level and closes the WebSocket frame. No reconnect mechanism exists (see §7.1 asymmetry note).
- **gRPC sink**: Logs `"gRPC client disconnected (buffer overflow)"` at WARN level and closes the gRPC stream. gRPC clients must implement their own reconnect with `last_event_id` in the `Subscribe` request.

The `max_buffer_size` default of 10,000 events provides burst tolerance for short consumer disconnects. At typical CDC throughput (hundreds to low thousands of events per second), this buffers approximately 10-60 seconds of events. Operators with spiky workloads or frequent consumer disconnects should increase this value, trading off per-sink memory consumption for burst tolerance.

### 8.3 `tap dev` Behavior Change

| | v0.1.0 | v0.2.0 |
|---|---|---|
| Default mode | Embedded (in-process) | Sidecar (starts `tap-sidecar` subprocess on `:9911`) |
| Flag to preserve v0.1.0 behavior | Not applicable | `tap dev --embedded` |
| Port used by default | None (in-process) | `9911` |

**Migration**: v0.1.0 users running `tap dev` in scripts or CI must add `--embedded` to maintain the same behavior. No data migration is required; state store format is unchanged.

**`tap dev` subprocess lifecycle** (sidecar mode):

- `tap dev` (without `--embedded`) starts `tap-sidecar` as a child process and holds its PID.
- On `tap dev` exit (SIGINT, SIGTERM, or process death), it sends SIGTERM to the sidecar child, waits up to 5 seconds for graceful shutdown, then sends SIGKILL if the sidecar has not exited.
- **Orphan handling**: If `tap dev` is killed without the sidecar child receiving SIGTERM (e.g., `kill -9`), the sidecar may survive as an orphan. On the next `tap dev` invocation, it checks whether `localhost:9911` is already in use; if so, it logs a warning and exits with a clear error message rather than silently starting a second sidecar.
- **Loopback binding check**: The sidecar HTTP/gRPC servers bind to `127.0.0.1` (loopback) by default, not `0.0.0.0`. Operators wanting remote access must explicitly set `sidecar.bind_address = "0.0.0.0"` in `tap.toml` AND acknowledge a startup warning: `"Sidecar bound to 0.0.0.0 — this exposes the CDC stream to any host that can reach <port>. Set up firewall rules or use a reverse proxy with auth."`. This prevents accidental exposure of the unauthenticated HTTP API in dev environments.
- **Port-in-use handling**: If `9911` is already bound by another process (not a previous sidecar orphan), `tap dev` exits with an actionable error: `"Port 9911 is in use. Use --port <PORT> to choose a different port or --embedded to skip sidecar mode."`.
- **First-run warning**: On the first `tap dev` invocation in a project directory where no `sidecar.port` is configured, `tap dev` logs a notice: `"Running in sidecar mode (default in v0.2.0). Pass --embedded to run in-process (v0.1.0 behavior)."`.
- **v0.1.0 `:9911` status page**: The v0.1.0 embedded mode had no HTTP status page. The sidecar's `GET /health` at `:9911` is the replacement and superset.

### 8.4 SQLite State Store Schema Migration

On first startup with v0.2.0, `tap-core` runs an automatic migration on the existing `state.db`. The migration bumps `schema_version` from 1 to 2.

**v0.1.0 `offsets` actual schema** (ground truth):

```
id, committed_lsn (TEXT UNIQUE), tx_id, ts_ms, sequence, is_final, instance_id, created_at
Indexes: idx_offsets_lsn (committed_lsn), idx_offsets_sequence (sequence), idx_offsets_final (is_final WHERE is_final=1)
```

**Migration steps**:

1. Write backup: `state.db` → `state.db.v0.1.0.bak`.
2. Rename column `committed_lsn` → `position` on the `offsets` table. The `UNIQUE` constraint, `tx_id`, `sequence`, `is_final`, `instance_id`, `created_at`, `AUTOINCREMENT`, and all existing rows are preserved.
3. Rename index `idx_offsets_lsn` → `idx_offsets_position` (on the new `position` column). Indexes `idx_offsets_sequence` and `idx_offsets_final` are unchanged.
4. Add column `adapter TEXT NOT NULL DEFAULT 'postgres'` to `offsets`. Existing rows receive the default `'postgres'`.
5. Rename table `skipped_lsns` → `skipped_positions`. Rename column `lsn` → `position` within that table. Rename index `idx_skipped_lsns_lsn` → `idx_skipped_positions_position`.
6. Add `snapshot_chunks` table (new in v0.2.0 — see §10.2).
7. Ensure `UNIQUE(sequence)` constraint on the `offsets` table: existing v0.1.0 offsets tables may lack this constraint. The migration adds it via `CREATE UNIQUE INDEX IF NOT EXISTS idx_offsets_sequence_unique ON offsets(sequence)`. We use `CREATE UNIQUE INDEX` (not a table-level `UNIQUE` constraint) because the table already exists in v0.1.0 and adding a table-level constraint to an existing table would require a destructive `ALTER TABLE`. The index achieves identical enforcement with no table rewrite. For new v0.2.0 databases, the table-level `UNIQUE(sequence)` constraint in §10.2's DDL provides the same guarantee at table creation time. If duplicates exist, the migration fails with an actionable error pointing the operator to a deduplication script.
8. Ensure `schema_version` table exists; insert row `(version=2, description='v0.2.0 migration')`.

**Resume ordering**: After migration, Tap resumes from the position stored in `offsets` ordered by `sequence` (the monotonically increasing column preserved from v0.1.0). It does NOT use `MAX(position)` string comparison, because `sequence` is the authoritative ordering key regardless of adapter type.

The migration is idempotent: running it on an already-migrated database is a no-op. If it fails (e.g., due to a locked database), `tap-core` exits with a clear error message. The backup file `state.db.v0.1.0.bak` is written before any schema changes begin.

**PID lock file invariant**: On startup, `tap-core` creates `.tap/state.db.lock` containing its PID and acquires an OS-level `flock()` (or equivalent advisory lock). If the lock file already exists and the PID it contains is still alive, the process exits with a fatal error: `"state.db is already in use by pid <PID>".` If the PID is stale (process no longer running), the lock file is overwritten. This invariant prevents two `tap-core` instances from sharing the same state store, which would corrupt the `offsets`, `snapshot_chunks`, and `snapshots` tables. The lock is released on graceful shutdown; on hard crash it is released by the OS when the process terminates. The lock file path is `{state_db_path}.lock` — always alongside the state database.

**SQLite durability configuration**: On first open (including after migration), `tap-core` sets the following PRAGMAs on the state store:

| PRAGMA | Value | Rationale |
|--------|-------|-----------|
| `journal_mode` | `WAL` | Write-ahead logging enables concurrent reads during checkpoint flushes. Without WAL, a checkpoint-write blocks all reads (and vice versa). |
| `synchronous` | `NORMAL` | In WAL mode, `NORMAL` is crash-safe and ~2x faster than `FULL`. Transactions are committed before the WAL syncs, so a crash may lose the last committed transaction but will not corrupt the database. |
| `busy_timeout` | `5000` | Wait up to 5 seconds if the database is locked before returning `SQLITE_BUSY`. This prevents transient contention from masking as a fatal error. |

**`integrity_check` on startup**: After the migration (or on first open for fresh databases), `tap-core` runs `PRAGMA integrity_check`. If the check returns any row other than `"ok"`, the process exits with a FATAL error: `"state.db integrity check failed: <message>"`. The backup from step 1 of the migration (if present) is left in place for operator recovery. This detects corruption early rather than manifesting as cryptic errors during capture.

**Database rebuild procedure**: Operators can rebuild the state store from scratch by deleting the `.tap/state.db` file (and the `.tap/state.db.lock` file if present) while the capture is stopped. On next startup, `tap-core` creates a fresh state store and initiates a full re-snapshot. The `tap.toml` config is not affected. This is the recommended recovery procedure for a corrupted state store.

### 8.5 Downgrade Path

v0.2.0 schema and state store changes are designed to be **forward-compatible only**: there is no downgrade migration. If you need to revert to v0.1.0 after running v0.2.0:

1. **Stop** the v0.2.0 capture gracefully (ensures the final checkpoint is flushed).
2. **Restore** the state store backup created during migration: `cp state.db.v0.1.0.bak state.db`.
3. **Roll back** the `tap` binary to v0.1.0.
4. **Restart** capture.

**Limitations**:
- The backup file `state.db.v0.1.0.bak` is written once during the first v0.2.0 startup. If the state store changes after migration (new `offsets` rows, `snapshot_chunks` entries), the backup reflects the snapshot of the data at migration time, not the current state. Downgrading to the backup will lose any offset progress, snapshot progress, and checkpoint data accumulated while running v0.2.0. A full re-snapshot will be required after the downgrade.
- If the backup file has been deleted or corrupted, downgrade is not possible. The operator must erase the state store (`rm .tap/state.db`) and accept a full re-snapshot on the v0.1.0 binary.
- v0.2.0 config files (`tap.toml`) using new v0.2.0 parameters (e.g., `[[transforms]]` blocks, `sidecar.*` settings) will cause v0.1.0 to fail parsing. A v0.1.0-compatible config must be provided.
- Downgrade from v0.2.0 to v0.1.0 is only supported from the same minor version (i.e., 0.2.0 → 0.1.0). Downgrade across patch versions is not guaranteed.

### 9.4 Configuration Precedence

Tap configuration is resolved from three sources, in descending order of precedence:

1. **CLI flags** (highest): Command-line flags such as `--port`, `--log-level`, `--config` always take precedence over all other sources.
2. **Environment variables**: For every config key `section.key`, the environment variable `TAP_SECTION_KEY` is checked (e.g., `TAP_SIDECAR_PORT=9922` overrides `sidecar.port` in the config file). Environment variables use uppercase with underscores; dot-separated config paths are converted by replacing `.` with `_` and uppercasing.
3. **Config file** (lowest): The `tap.toml` config file (default path: `.tap/config.toml`, overridable via `--config`). Only keys explicitly set in the file are applied — unset keys use their built-in defaults.

**Resolution rule**: At each precedence level, only explicitly provided values override the level below. If a key is not set at any level, its built-in default is used. This means `--port 9922` on the CLI overrides `sidecar.port = 9911` in `tap.toml`, which in turn overrides the built-in default of 9911. Setting `TAP_SIDECAR_PORT=9922` overrides `tap.toml` but is itself overridden by `--port 9923`.

### 9.5 Config File Discovery

Config file discovery is independent of value precedence (see §9.4). The file to load is determined by this search order:

1. **`--config <PATH>` flag** (highest): If specified, that exact path is used.
2. **`.tap/config.toml`** (project-scoped): Checked next if no `--config` flag.
3. **`./tap.toml`** (project root): Fallback for SDK default path compatibility.
4. **`TAP_CONFIG_PATH` environment variable**: Final fallback. Note: while env var values take precedence over config file values (per §9.4), `TAP_CONFIG_PATH` selects *which file to load*, not which value wins. The values inside that file are still the lowest-precedence level.

The first file found is used; there is no merging across multiple config files. If no config file is found at any location, the sidecar starts with built-in defaults (all config keys use their documented defaults).

**Unknown keys**: Unknown keys in `tap.toml` or environment variables emit a WARN-level log at startup but do not prevent startup. This accommodates forward-compatible config files shared across versions.

The DDL below is the post-migration state of `state.db` after the v0.2.0 migration runs (see §8.4). It is derived from the v0.1.0 DDL by renaming `committed_lsn`→`position` on `offsets`, adding `adapter`, renaming `skipped_lsns`→`skipped_positions`, and adding `snapshot_chunks`.

```sql
-- ============================================================
-- Offset / position tracking (migrated from v0.1.0 offsets)
-- committed_lsn renamed to position; adapter column added.
-- All v0.1.0 constraints (UNIQUE, sequence, is_final) preserved.
-- ============================================================
CREATE TABLE IF NOT EXISTS offsets (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    -- "0/16B37428" (Postgres LSN) or "mysql-bin.000123:4567890" (MySQL binlog coordinate)
    position        TEXT    NOT NULL UNIQUE,
    tx_id           TEXT    NOT NULL,
    ts_ms           INTEGER NOT NULL,
    -- Monotonically increasing; used for resume ordering (ORDER BY sequence DESC LIMIT 1)
    sequence        INTEGER NOT NULL,
    UNIQUE(sequence),
    -- 1 on clean shutdown checkpoint, 0 on mid-stream checkpoint
    is_final        INTEGER NOT NULL DEFAULT 0,
    instance_id     TEXT,
    -- Which adapter wrote this row (forward-compat; one adapter per process)
    adapter         TEXT    NOT NULL DEFAULT 'postgres',
    created_at      TEXT    NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_offsets_position ON offsets(position);
CREATE INDEX idx_offsets_sequence ON offsets(sequence);
CREATE INDEX idx_offsets_final ON offsets(is_final) WHERE is_final = 1;

-- ============================================================
-- Skipped positions (renamed from v0.1.0 skipped_lsns)
-- ============================================================
CREATE TABLE IF NOT EXISTS skipped_positions (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    position        TEXT    NOT NULL,   -- LSN or binlog coordinate that failed
    tx_id           TEXT,
    error_message   TEXT    NOT NULL,
    raw_data        TEXT,
    created_at      TEXT    NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_skipped_positions_position ON skipped_positions(position);

-- ============================================================
-- Snapshot progress (unchanged from v0.1.0 structure)
-- ============================================================
CREATE TABLE IF NOT EXISTS snapshots (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    table_name      TEXT    NOT NULL,
    snapshot_id     TEXT    NOT NULL,
    rows_count      INTEGER NOT NULL DEFAULT 0,
    status          TEXT    NOT NULL DEFAULT 'in_progress',
    started_at      TEXT    NOT NULL DEFAULT (datetime('now')),
    completed_at    TEXT,
    error_message   TEXT,
    snapshot_lsn    TEXT    NOT NULL,
    UNIQUE(table_name, snapshot_id)
);

CREATE INDEX idx_snapshots_table ON snapshots(table_name);
CREATE INDEX idx_snapshots_status ON snapshots(status);

-- ============================================================
-- Parallel snapshot chunk progress (NEW in v0.2.0)
-- ============================================================
CREATE TABLE IF NOT EXISTS snapshot_chunks (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    snapshot_id     INTEGER NOT NULL REFERENCES snapshots(id),
    chunk_index     INTEGER NOT NULL,
    pk_start        TEXT    NOT NULL,   -- inclusive lower bound (native type serialized as text)
    pk_end          TEXT    NOT NULL,   -- exclusive upper bound
    last_pk         TEXT,               -- last pk value emitted; used for resume
    rows_emitted    INTEGER NOT NULL DEFAULT 0,
    completed       INTEGER NOT NULL DEFAULT 0,
    completed_at    TEXT,
    UNIQUE(snapshot_id, chunk_index)
);

-- ============================================================
-- Schema version tracking (NEW in v0.2.0)
-- v0.1.0 schema_version starts at 1; migration brings it to 2.
-- ============================================================
CREATE TABLE IF NOT EXISTS schema_version (
    version         INTEGER NOT NULL PRIMARY KEY,
    applied_at      TEXT    NOT NULL DEFAULT (datetime('now'))
);

INSERT OR IGNORE INTO schema_version (version) VALUES (2);
```

**`adapter` column**: Records `'postgres'` or `'mysql'` per row. Present for forward-compatibility with shared state stores across deployments. A single `tap-core` process runs exactly one adapter (see §3.5); this column does not enable multi-source capture within one process.

## 11. Performance Targets

### 11.1 Parallel Snapshot Throughput

| Scenario | Target | Measurement Conditions |
|---|---|---|
| 10M-row table, 4 workers | < 5 minutes to full snapshot | Postgres 15, m5.xlarge EC2 (4 vCPU, 16 GB RAM), row size ~200 bytes, SSD storage |
| 10M-row table, sequential (v0.1.0 baseline) | ~30+ minutes | Same conditions |
| 50M-row table, 4 workers | < 25 minutes | Same conditions |

Throughput target: **≥ 33,000 rows/second** across all parallel workers combined for a 10M-row table in ≤ 5 minutes.

These targets assume the bottleneck is database I/O, not Tap processing. If the database cannot sustain the read rate, the snapshot will take longer. Users with constrained read IOPS should reduce `max_snapshot_workers`.

### 11.2 Transform Pipeline Overhead Budget

Transforms add latency to the hot path between raw event decode and sink delivery. The overhead budget is:

| Transform type | Per-event overhead target (p99) |
|---|---|
| No transforms (baseline) | — |
| Single `filter` transform | < 0.5 ms |
| Single `map` transform | < 1 ms |
| Single `mask` transform (config-only) | < 0.1 ms |
| Three-stage pipeline (filter + map + mask) | < 2 ms |

These targets are measured on an m5.xlarge EC2 instance (4 vCPU). The QuickJS WASM context creation dominates overhead; future optimization may cache and reuse contexts across events.

### 11.3 Sidecar Mode Overhead

| Metric | Target |
|---|---|
| SSE event delivery latency (local loopback) | < 5 ms additional vs. embedded mode |
| WebSocket event delivery latency (local loopback) | < 5 ms additional vs. embedded mode |
| Sidecar idle memory footprint | < 30 MB RSS |
| Sidecar under full load (10,000 events/sec) | < 100 MB RSS |

### 11.4 End-to-End Capture Latency (v0.2.0)

| Metric | Target |
|---|---|
| Single-row Postgres insert → SSE delivery (embedded) | < 100 ms (p99) — unchanged from v0.1.0 |
| Single-row MySQL insert → SSE delivery (embedded) | < 150 ms (p99) |
| Single-row Postgres insert → SSE delivery (sidecar) | < 200 ms (p99) |

## 14. Open Questions and Risks

| ID | Question | Risk if Unresolved | Status |
|---|---|---|---|
| 14.1 | **MySQL crate selection**: Use `mysql_async`'s built-in `binlog` feature or a dedicated binlog crate? `mysql_binlog` (EasyPost, archived Apr 2026) is a parser-only, dead project. `mysql-binlog-connector-rust` (apecloud) is actively maintained but an unnecessary extra dependency. `mysql_async` (v0.37, with `binlog` feature) already implements the `COM_BINLOG_DUMP` handshake, event streaming, GTID, and checksum handling natively. | Medium (resolved). `mysql_async + binlog` feature selected; removes an empty dependency decision. | **Resolved** — use `mysql_async` v0.37 with `features = ["binlog"]`. No dedicated binlog crate needed. |
| 14.2 | **QuickJS WASM embedding strategy**: **Resolved — wasmtime** (v0.2.0 final). Decision: embed QuickJS compiled to WASM, hosted inside `wasmtime`. Rationale: (1) epoch-based interruption is reliable and well-documented in `wasmtime` — the 100ms timeout in §4.3.5 can be enforced precisely; (2) `wasmtime` is actively maintained by the Bytecode Alliance; (3) `quickjs-wasm-rs` has had multiple extended maintenance gaps and we do not want transform correctness to depend on it. Cost accepted: ~6 MB additional binary size total from `wasmtime` runtime + QuickJS WASM blob. This adds approximately 60:40 split — ~3.6 MB to the `tap-sidecar` binary and ~2.4 MB to the embedded mode (`tap-core` linked into TS/Python SDKs). Both figures are combined ~6 MB total. For SDKs, this is a 1.5–2x size increase — acceptable for v0.2.0 but worth re-evaluating in v0.3.0 if SDK size becomes a concern. | Medium (resolved) | **Resolved** — implementation in progress |
| 14.3 | **gRPC port assignment** (resolved): Choose between sharing port `9911` via HTTP/2 protocol negotiation (h2c) or using a separate port. Sharing port would require `axum` to multiplex HTTP/1.1 (SSE) and HTTP/2-only (gRPC) traffic on the same socket. The dedicated-port option was selected for simpler implementation and cleaner deployment. | Low (resolved) | **Resolved** — dedicated port `9912` (see §7.2) |
| 14.4 | **Python asyncio bridge**: Use `pyo3-asyncio` (which pins to specific `tokio` and `pyo3` version combinations) or implement a bespoke channel bridge? `pyo3-asyncio` version compatibility has historically been fragile. | Medium: asyncio integration bugs are the most common failure mode in pyo3 projects. | **Resolved** — bespoke channel bridge preferred (see §6.2) |
| 14.5 | **MySQL consistent snapshot locking**: `FLUSH TABLES WITH READ LOCK` acquires a global lock that blocks all writes for the duration of worker connection setup (not data scan — see §4.2.3). For `max_snapshot_workers = 4` on a typical local or in-region network, the window is 50-200ms; on a high-latency link or with higher `max_snapshot_workers`, it may exceed 5 seconds. The lock duration is observable via the `tap_snapshot_lock_held_ms` metric (see §5.2.3). | Medium — addressed (metric-backed). | **Accepted** for v0.2.0 with metric-backed observability; per-table lock alternative deferred to v0.3.0 |
| 14.6 | **`snapshot` field serialization change**: Changing `snapshot` from `undefined` (omitted) to `null` (explicit) in the JSON envelope is a behavioral change for consumers using `"snapshot" in event.source` checks in JavaScript. Is this change worth the consistency gain? | Low: the `skip_serializing_if` attribute can be retained for the `snapshot` field to avoid the change entirely. | **Resolved** — `skip_serializing_if` retained (see §8.1) |
| 14.7 | **Transform fail-open vs. fail-closed default**: The spec proposes fail-open (pass event through on transform error) as the default. Some users may prefer fail-closed (drop event on error) as the safer default for privacy transforms. | Low: configurable via `transform_on_error`. Needs decision on which default is safer for the common case. | **Resolved** — filter/map fail-open, mask fail-closed (see §4.3.5) |
| 14.8 | **Windows support for Python SDK**: No Windows wheel is planned for v0.2.0. Python data engineers on Windows (or WSL) will need to build from source. Is this acceptable for v0.2.0? | Low: Python data engineering workloads are predominantly Linux/macOS. Windows support can be added in a patch. | **Resolved** — accepted limitation for v0.2.0. Deferred indefinitely; revisit when 3+ distinct user requests materialize from native Windows (non-WSL) users. |
| 14.9 | **`sidecar.unix_socket` default path**: `/tmp/tap.sock` may conflict with other processes on shared systems. A project-scoped default (e.g., `.tap/tap.sock`) avoids collisions but requires the `.tap/` directory to exist. | Low: configurable. Default choice is a UX decision. | **Resolved** — default changed to `.tap/tap.sock`. Rationale: (1) `.tap/` already exists for state.db/config; socket there is consistent. (2) `/tmp/` is world-readable on Unix — `.tap/` inherits project permissions. (3) Prevents cross-project socket conflicts. Socket umask set to 0700. |
| 14.10 | **TS SDK sidecar reconnection**: When the sidecar restarts, the TS SDK in sidecar mode reconnects with exponential backoff + jitter and resumes from the last seen `event.id`. **Policy (resolved for v0.2.0)**: (1) Initial reconnect attempt after 100ms; (2) Backoff doubles per failed attempt, capped at 30s; (3) Jitter: ±20% of the backoff value (e.g., 100ms ± 20ms, then 200ms ± 40ms); (4) Resume via `GET /events?last_event_id=<id>` (SSE) or equivalent WebSocket message — the sidecar replays buffered events from an in-memory ring buffer (`max_buffer_size`, default 10,000 events; effective retention ≈ `max_buffer_size / events_per_second`). This buffer is ephemeral: a sidecar process restart clears it. After a sidecar restart, the SDK resumes from the last committed `position` in the `offsets` table rather than from the in-memory buffer. The replay buffer exists to absorb short disconnects; the offsets table provides durable resume across process restarts. (5) After 10 consecutive failed reconnects, the SDK emits an `error` event and stops attempting; the application must restart. The connection state is exposed via `TapClient.status` (`connecting` | `streaming` | `reconnecting` | `replay` | `error`). | Medium (resolved) | **Resolved** — implementation matches the policy above |
