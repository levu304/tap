## 1. Overview

Tap v0.1.0 — codename **"Foundation"** — is the first release of a next-generation Change Data Capture (CDC) platform. This release proves the core engine works: it captures row-level changes from a single PostgreSQL database via logical replication slots, serializes them into a standardized `ChangeEvent` envelope, streams them to consumers over HTTP/2 Server-Sent Events (SSE), and persists checkpoint state in a local SQLite store for idempotent delivery. Developers interact with Tap through a TypeScript SDK (powered by `napi-rs` for native performance) and a Rust CLI for project scaffolding, capture management, schema introspection, and local development. This is the minimal viable foundation upon which all future modes (sidecar, cluster), languages (Python, Go, Elixir, Java, C#), transports (Kafka, Redis, NATS, gRPC, WebSocket), and features (transforms, schema registry, WASM) will be built.

```
│           START_REPLICATION SLOT ... LOGICAL ...
│           → Decode WAL records via pgoutput/wal2json
```

### 3.3 Data Flow Summary

1. **Startup**: CLI or SDK initializes `tap-core`, loads config, opens/creates SQLite state store.
2. **Connection**: Rust core connects to Postgres via `libpq` (or `tokio-postgres`), validates replication slot/publication, or creates them if missing.
3. **Checkpoint Recovery**: Reads last committed LSN from SQLite `offsets` table. If zero, begins snapshot.
4. **Snapshot**: Acquires consistent snapshot via `pg_export_snapshot()`, iterates tables, emits `op: 'r'` events, tracks progress.
5. **Transition**: Closes snapshot, starts logical replication at the snapshot's LSN.
6. **Streaming**: Decodes WAL changes (insert/update/delete) into `ChangeEvent` structs, forwards to SSE sink.
7. **Checkpointing**: Every N events or M milliseconds, records current confirmed-flush LSN to SQLite.
8. **Consumption**: SSE HTTP server pushes events to connected clients. SDK `on('change')` fires callbacks.
9. **Shutdown**: SIGINT/SIGTERM → stop WAL reading → flush final LSN → close slot connection → exit.

## 4. Component Specifications

### 4.1 `tap-core` (Rust Crate)

**Location**: `crates/tap-core/`  
**Key dependencies**: `tokio-postgres` (async Postgres), `tokio` (async runtime), `rusqlite` (SQLite), `serde`/`serde_json` (serialization), `uuid` (event IDs), `tracing` (logging).

#### 4.1.1 Postgres Logical Decoding

**Plugin Support**:

| Plugin | Protocol | Default? | Status | When to Use |
|---|---|---|---|---|
| `pgoutput` | Built-in, pg 14+ | **Yes** | **First-class** | Standard Postgres logical replication. No extensions needed. |
| `wal2json` | Extension (wal2json) | No | **Experimental** | Older Postgres or JSON-formatted WAL preferred. Requires extension install. |

**`wal2json` Wire Format**:

`wal2json` emits a JSON object per transaction, not per change. Example:

```json
{
  "xid": 12345,
  "timestamp": "2026-05-31 12:00:00.123",
  "change": [
    {
      "kind": "insert",
      "schema": "public",
      "table": "users",
      "columnnames": ["id", "name", "email"],
      "columnvalues": [42, "Alice", "alice@example.com"],
      "oldkeys": null
    },
    {
      "kind": "update",
      "schema": "public",
      "table": "users",
      "columnnames": ["id", "name", "email"],
      "columnvalues": [42, "Alice", "alice@new.com"],
      "oldkeys": {"keynames": ["id"], "keyvalues": [42]}
    }
  ]
}
```

**`wal2json` Limitations (v0.1.0 Experimental)**:
- Does not emit `before` values for UPDATE/DELETE by default (only `new` values). The `before` field in `ChangeEvent` will be `null` for `wal2json`-captured events.
- No `Relation` messages — schema must be resolved independently.
- Reduced test coverage in v0.1.0; not all features tested.
- Requires extension installation (`CREATE EXTENSION IF NOT EXISTS wal2json;`) — not available on all managed Postgres providers.

**Connection Management**:

- Use `tokio-postgres` with `replication` connection mode.
- Connection string from config (e.g., `postgres://user:pass@localhost:5432/db?replication=database`).
- On startup:
  1. Connect to Postgres.
  2. Validate or create replication slot (`tap_slot` by default).
  3. Validate or create publication (`tap_publication` by default).
  4. If publication includes no tables, default to all tables in the database.
- On shutdown:
  1. Send `FlushLSN` to Postgres.
  2. Optionally drop replication slot (configurable, default: retain for resume).
  3. Close connection.

**`START_REPLICATION` Protocol**:

```rust
// Pseudocode for logical replication stream
let slot_name = config.slot_name; // default: "tap_slot"
let publication = config.publication; // default: "tap_publication"
let start_lsn = state_store.get_last_checkpoint().unwrap_or(LSN::ZERO);

let stream = postgres_client
    .start_replication(slot_name, publication, start_lsn, plugin)
    .await?;

// Each WAL record yields a LogicalReplicationMessage
while let Some(msg) = stream.next().await {
    match msg {
        Message::Begin { xid, lsn, timestamp } => { /* start tx */ }
        Message::Insert { relation_id, row_data } => { /* build event */ }
        Message::Update { relation_id, old_row, new_row } => { /* build event */ }
        Message::Delete { relation_id, old_row } => { /* build event */ }
        Message::Commit { xid, lsn, timestamp } => { /* finalize & flush */ }
        Message::Relation { relation_id, schema, table, columns } => { /* cache schema */ }
        Message::Type { .. } => { /* ignore type mapping */ }
        Message::Origin { .. } => { /* ignore origin */ }
        _ => { /* skip unknown */ }
    }
}
```

**LSN Tracking**: Stored in SQLite `offsets` table. On every commit boundary, the confirmed-flush LSN is persisted. On resume, the core starts from the last persisted LSN.

**Configuration Parameters**:

| Parameter | Default | Description |
|---|---|---|
| `slot_name` | `"tap_slot"` | Replication slot name |
| `publication` | `"tap_publication"` | Publication name |
| `tables` | `[]` (all tables) | Specific tables to capture (e.g., `["public.users", "public.orders"]`) |
| `plugin` | `"pgoutput"` | Decoding plugin: `pgoutput` or `wal2json` |
| `max_batch_size` | `1000` | Max events per batch before flush |
| `flush_interval_ms` | `1000` | Max milliseconds between checkpoints |

#### 4.1.2 ChangeEvent Envelope

Each change event is a JSON object serialized from a Rust struct:

```rust
use serde::{Serialize, Deserialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ChangeEvent {
    /// Operation type: c=create, u=update, d=delete, r=read (snapshot)
    pub op: String,

    /// Row state before the change (null for inserts)
    pub before: Option<serde_json::Value>,

    /// Row state after the change (null for deletes)
    pub after: Option<serde_json::Value>,

    /// Source metadata
    pub source: SourceMetadata,

    /// Timestamp when the event was captured (epoch ms)
    pub ts_ms: u64,

    /// Globally unique event ID for idempotent consumer deduplication
    pub id: String,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SourceMetadata {
    /// Database name
    pub db: String,

    /// Schema name
    pub schema: String,

    /// Table name
    pub table: String,

    /// Postgres LSN (e.g., "0/16B37428")
    pub lsn: String,

    /// Transaction ID (e.g., "12345")
    pub tx_id: String,

    /// Timestamp from the source (Postgres commit timestamp, epoch ms)
    pub ts_ms: u64,

    /// Whether this event came from a snapshot
    #[serde(skip_serializing_if = "Option::is_none")]
    pub snapshot: Option<bool>,
}
```

**Example Events**:

```json
// Insert (create)
{
  "op": "c",
  "before": null,
  "after": {"id": 42, "name": "Alice", "email": "alice@example.com"},
  "source": {"db": "mydb", "schema": "public", "table": "users", "lsn": "0/16B37428", "tx_id": "12345", "ts_ms": 1717200000123},
  "ts_ms": 1717200000150,
  "id": "0/16B37428:12345"
}

// Update
{
  "op": "u",
  "before": {"id": 42, "name": "Alice", "email": "alice@old.com"},
  "after": {"id": 42, "name": "Alice", "email": "alice@new.com"},
  "source": {"db": "mydb", "schema": "public", "table": "users", "lsn": "0/16B37430", "tx_id": "12346", "ts_ms": 1717200000500},
  "ts_ms": 1717200000530,
  "id": "0/16B37430:12346"
}

// Delete
{
  "op": "d",
  "before": {"id": 42, "name": "Alice", "email": "alice@example.com"},
  "after": null,
  "source": {"db": "mydb", "schema": "public", "table": "users", "lsn": "0/16B37438", "tx_id": "12347", "ts_ms": 1717200001000},
  "ts_ms": 1717200001020,
  "id": "0/16B37438:12347"
}

// Snapshot row (read)
{
  "op": "r",
  "before": null,
  "after": {"id": 1, "name": "Bob"},
  "source": {"db": "mydb", "schema": "public", "table": "users", "lsn": "0/16B37000", "tx_id": "0", "ts_ms": 1717200000000, "snapshot": true},
  "ts_ms": 1717200000000,
  "id": "snap:public.users:1"
}
```

**Idempotent Event IDs**:

- **Streaming events**: `"{lsn}:{tx_id}"` — deterministic, globally unique per Postgres instance. Consumers deduplicate by storing processed event IDs.
- **Snapshot events**: `"snap:{schema}.{table}:{pk_value}"` — deterministic within a snapshot run. If a snapshot is restarted, events carry the same IDs (idempotent). For composite primary keys, the format uses explicit key:value pairs: `"snap:public.order_items:user_id=1:order_id=2"`. This avoids ambiguity with single-column keys.

> **Note on delivery semantics**: Tap provides deterministic event IDs and checkpointed delivery, but consumers must implement deduplication (e.g., by tracking processed IDs). This is **idempotent delivery**, not server-side exactly-once. A built-in `TapEventStore` helper for consumer-side dedup will be provided in a future SDK version. See Section 8.1 for guidance on handling re-delivered events.

#### 4.1.3 WAL Position Tracking (SQLite State Store)

**State Store Schema** — see Section 7 for full DDL.

The state store lives at `.tap/state.db` (relative to the project root) or a configurable path.

**Checkpoint Behavior**:

- **Batch checkpointing (default)**: Events accumulate in memory; the commit LSN is flushed to the `offsets` table every `flush_interval_ms` (default 1000ms) or every `max_batch_size` events (default 1000), whichever comes first. This provides deterministic event IDs and consumer dedup at much higher throughput than per-commit flushing.
- **On graceful shutdown**: A final flush writes the last processed LSN.
- **Resume**: On next startup, the core reads `MAX(committed_lsn)` from the `offsets` table. If non-zero, `START_REPLICATION` resumes from that LSN.

**Snapshot Progress Tracking**:

The `snapshots` table tracks which tables have been fully snapshot. If a snapshot is interrupted (e.g., crash mid-snapshot), the next run can skip already-completed tables.

#### 4.1.4 Sequential Snapshotting

**Algorithm**:

1. Connect to Postgres.
2. Call `pg_export_snapshot()` to obtain a transaction-consistent snapshot ID.
3. For each configured table (ordered by schema + name):
   a. Execute `SELECT * FROM {schema}.{table} ORDER BY primary_key`.
   b. For each row, emit a `ChangeEvent` with `op: "r"` and `snapshot: true`.
   c. Periodically checkpoint the last emitted primary key value in the `snapshots` table.
4. Release the snapshot via `SET TRANSACTION SNAPSHOT` cleanup.
5. Mark snapshot as complete in the `snapshots` table.
6. Begin streaming from the LSN obtained at snapshot export time.

**Key Design Decisions**:

- **Single-threaded**: Tables processed sequentially. Parallelism deferred to v0.2.0.
- **Consistent point-in-time**: `pg_export_snapshot()` guarantees snapshot and streaming start from the same database state.
- **No gap**: The snapshot's LSN is saved. The streaming phase begins at that LSN, ensuring no events are missed between snapshot end and streaming start.
- **Primary key requirement**: Tables without a primary key are logged as a warning and processed but cannot resume partial snapshots.

**Edge Cases**:

- **Empty tables**: Emit no events (no-op), record as complete, proceed.
- **Tables without PK**: Warn via log, use `ctid` for checkpoint if needed, but full re-snapshot if interrupted.
- **Schema changes during snapshot**: If DDL occurs (ALTER TABLE), `pg_export_snapshot()` provides a consistent view. DDL after snapshot but before streaming is captured via WAL. In v0.1.0, schema evolution events are collected but not stored in a registry.

#### 4.1.5 Error Handling

| Scenario | Behavior |
|---|---|
| **Connection loss to Postgres** | Automatic reconnection with exponential backoff: 1s → 2s → 4s → 8s → max 60s. After 10 retries, exit with error code. |
| **Replication slot lost** | Log fatal error, exit. User must re-run `tap init` to recreate the slot. |
| **Publication missing** | Log warning, attempt to recreate, fail if insufficient permissions. |
| **WAL record decode failure** | Log error with LSN position, skip record, continue. The skipped LSN is recorded in a `skipped_lsns` table for manual audit. |
| **SQLite corruption** | On checksum mismatch, attempt to restore from `.tap/state.db.bak`. If backup is also corrupt, exit with error and instructions. |
| **SSE client disconnect** | Buffer up to `max_buffer_size` events (configurable, default 10,000). If buffer exceeds limit, pause WAL reading until consumer reconnects. |
| **Graceful shutdown** | SIGINT/SIGTERM → flush LSN → close connections → exit cleanly (≤ 1 second). See Sections 5 (Step 9) and 8.4. |

**Reconnection Configuration**:

```rust
pub struct ReconnectConfig {
    pub max_retries: u32,          // default: 10
    pub initial_backoff_ms: u64,   // default: 1000
    pub max_backoff_ms: u64,       // default: 60000
    pub jitter: bool,              // default: true (adds ±20% randomness)
}
```

##### `tap init`

```
tap init [OPTIONS]

Scaffold a new Tap project in the current directory.

Options:
  -n, --name <name>          Project name (default: current directory name)
  -d, --db <CONNECTION_STR>  Postgres connection string
  -s, --slot <SLOT>          Replication slot name (default: "tap_slot")
  -p, --publication <PUB>    Publication name (default: "tap_publication")
  -t, --tables <TABLES>...   Tables to capture (default: all)
  -o, --output <DIR>         Output directory (default: ".")
      --plugin <PLUGIN>      Decoding plugin: pgoutput | wal2json (default: pgoutput)
      --force                Overwrite existing files without prompting
```

**Behavior**:

1. Creates `.tap/` directory structure.
2. Generates `.tap/config.toml` with provided or default values.
3. Connects to Postgres, validates credentials, tests replication permissions.
4. Creates replication slot and publication if they don't exist.
5. Inspects database schema and generates TypeScript type definitions.
6. Creates `tap.ts` (or `tap.js`) entry point with boilerplate code.
7. Creates `.gitignore` with `.tap/state.db` entries.
8. Outputs success message with next steps.

**Scaffolded Structure**:

```
my-tap-project/
├── .tap/
│   ├── config.toml
│   └── types/          # Auto-generated TS types
│       └── schema.d.ts
├── tap.ts              # Boilerplate entry point
├── package.json        # If not present, generated
└── .gitignore
```

```
  -c, --config <PATH>       Config file path (default: .tap/config.toml)
  -d, --db <CONNECTION_STR> Override Postgres connection string
  -s, --slot <SLOT>         Override replication slot name
  -p, --publication <PUB>   Override publication name
  -t, --tables <TABLES>...  Filter tables (comma-separated)
      --from-lsn <LSN>      Start from specific LSN (overrides state store)
      --snapshot            Force full re-snapshot
      --no-snapshot         Skip snapshot even if first run
      --port <PORT>         SSE server port (default: 0 = random)
      --host <HOST>         SSE server bind address (default: 127.0.0.1)
```

**Behavior**:

1. Loads config from `.tap/config.toml` (or specified path).
2. Connects to Postgres.
3. Checks SQLite state store for existing checkpoint.
4. If no checkpoint → runs snapshot, then transitions to streaming.
5. If checkpoint exists → resumes streaming from saved LSN.
6. Starts an HTTP/2 SSE server on the specified port.
7. Outputs the SSE endpoint URL (e.g., `SSE endpoint: http://127.0.0.1:{assigned_port}/events` — port is dynamic when set to 0).
8. Runs until interrupted (SIGINT/SIGTERM) or fatal error.
9. On shutdown, flushes LSN and exits.

#### 4.3.1 `Tap` Class API

```typescript
import { Tap, type TapConfig, type ChangeEvent } from "tap-cdc";

// Configuration
interface TapConfig {
  /** Postgres connection string */
  connection: string;

  /** Replication slot name (default: "tap_slot") */
  slotName?: string;

  /** Publication name (default: "tap_publication") */
  publication?: string;

  /** Tables to capture (default: all tables in publication) */
  tables?: string[];

  /** Decoding plugin (default: "pgoutput") */
  plugin?: "pgoutput" | "wal2json";

  /** SSE server host (default: "127.0.0.1") */
  host?: string;

  /** SSE server port (default: 0 = random available port) */
  port?: number;

  /** Path to SQLite state store (default: ".tap/state.db") */
  statePath?: string;

  /** Maximum events per batch before flush (default: 1000) */
  maxBatchSize?: number;

  /** Max milliseconds between checkpoints (default: 1000) */
  flushIntervalMs?: number;
}

// Event types
type ChangeEventHandler = (event: ChangeEvent) => void;
type ErrorEventHandler = (error: Error) => void;
type StatusEventHandler = (status: CaptureStatus) => void;

interface CaptureStatus {
  state: "snapshot" | "streaming" | "paused" | "stopped";
  eventsCaptured: number;
  currentLsn: string;
  lagMs: number;
}

interface ChangeEvent {
  op: "c" | "u" | "d" | "r";
  before: Record<string, unknown> | null;
  after: Record<string, unknown> | null;
  source: {
    db: string;
    schema: string;
    table: string;
    lsn: string;
    tx_id: string;
    ts_ms: number;
    snapshot?: boolean;
  };
  ts_ms: number;
  id: string;
}

class Tap {
  constructor(config: TapConfig);

  /** Register a handler for change events */
  on(event: "change", handler: ChangeEventHandler): this;
  /** Register a handler for errors */
  on(event: "error", handler: ErrorEventHandler): this;
  /** Register a handler for status updates */
  on(event: "status", handler: StatusEventHandler): this;

  /** Remove a previously registered handler */
  off(event: "change" | "error" | "status", handler: Function): this;

  /** Start capturing. Returns the SSE URL. */
  start(): Promise<string>;

  /** Gracefully stop capturing. Flushes checkpoint. */
  stop(): Promise<void>;

  /** Pause capture (temporarily stop reading WAL). Buffers are retained. */
  pause(): Promise<void>;

  /** Resume capture after pause. */
  resume(): Promise<void>;

  /** Get current capture status */
  status(): CaptureStatus;

  /** Inspect database schema. Returns table schemas. */
  inspect(options?: InspectOptions): Promise<DatabaseSchema>;
}

// Usage example
const tap = new Tap({
  connection: "postgres://user:pass@localhost:5432/mydb",
  tables: ["public.users", "public.orders"],
});

tap.on("change", (event) => {
  console.log(`[${event.op}] ${event.source.table}:`, event.after);
});

tap.on("error", (err) => {
  console.error("Capture error:", err);
});

tap.on("status", (status) => {
  console.log(`Status: ${status.state}, ${status.eventsCaptured} events`);
});

await tap.start();
// Later...
await tap.stop();
```

#### 4.3.2.1 napi-rs Async Bridge Design

The napi-rs bridge is the critical FFI boundary between Rust's async runtime and the Node.js event loop. This section specifies how async events cross this boundary.

**Architecture**:

```
Rust Async Runtime (tokio)
┌──────────────────────────────────────────────────────┐
│  WAL Reader                                           │
│  ┌──────────┐    tokio::sync::mpsc (unbounded)       │
│  │ Decoder  │───►  Channel A: WAL → SSE serializer   │
│  └──────────┘    ┌────────────────────────────────┐   │
│                  │  SSE Server (HTTP/2, tokio)     │   │
│                  │  → TCP socket → external client  │   │
│                  └────────────────────────────────┘   │
│                       │                               │
│                 tokio::sync::mpsc (bounded)            │
│                       │                               │
│                  Channel B: WAL → napi-rs callback    │
│                  ┌────────────────────────────────┐   │
│                  │  napi::ThreadsafeFunction       │   │
│                  └──────────┬─────────────────────┘   │
└─────────────────────────────┼─────────────────────────┘
                              │
napi-rs Worker Thread         │
└─────────────────────────────┼─────────────────────────┘
                              ▼
Node.js Event Loop
┌──────────────────────────────────────────────────────┐
│  JS Callback Handler                                  │
│  tap.on('change', handler)(event)                     │
│  tap.on('error', handler)(error)                      │
│  tap.on('status', handler)(status)                    │
└──────────────────────────────────────────────────────┘
```

**Channel Strategy**:

| Channel | Type | Capacity | Purpose |
|---|---|---|---|
| A (Decoder → SSE) | `tokio::sync::mpsc` | Unbounded | WAL events → SSE HTTP serializer for external consumers |
| B (Decoder → napi-rs) | `tokio::sync::mpsc` | Bounded (default: `max_buffer_size` = 10,000) | WAL events → JS callback via `ThreadsafeFunction` |

**Backpressure**: When Channel B is full (JS consumer is slow), the WAL reader pauses decoding. The same `max_buffer_size` mechanism governs both SSE buffer (TCP backpressure) and napi-rs callback delivery. When the JS callback catches up and Channel B drains, WAL reading resumes.

**Thread Model**:

| Component | Runs On | Thread Safety |
|---|---|---|
| WAL Reader + Decoder | `tokio` async runtime (Rust thread pool) | Single `tokio` task, sequential WAL processing |
| SSE HTTP Server | `tokio` async runtime | Standard `tokio` task pool |
| napi-rs Callback Delivery | napi-rs worker thread | `ThreadsafeFunction` serializes calls into JS event loop |
| Node.js Event Loop | Main V8 thread | Single-threaded (JavaScript) |
| JS Handler Execution | Main V8 thread | Invoked by `ThreadsafeFunction` on JS event loop |

**Error Propagation**:

Rust-side errors (connection loss, decode failure, state store corruption) are captured as Rust `Error` types, converted to `napi::JsError`, and delivered to the JS `error` event handler via the same `ThreadsafeFunction` path. Unhandled JS errors in the callback are caught and logged, not silently swallowed.

**Dual Delivery Path**:

The napi-rs callback is the **primary delivery path** for SDK users. The SSE server runs in parallel for external HTTP consumers. See Section 4.3.3 for the delivery architecture.

#### 4.3.3 SSE Transport Format

The SDK supports a **dual delivery mechanism** — events reach consumers through two parallel paths:

1. **In-process callback** (primary for SDK users): The napi-rs `ThreadsafeFunction` delivers `ChangeEvent` structs directly to the JS `on('change')` handler, bypassing serialization and TCP overhead. This is the recommended path for embedded SDK usage — 0.5–2ms latency savings per event vs. SSE.
2. **HTTP/2 SSE server** (for external consumers): Runs on a dedicated thread, accepts standard `EventSource` clients, and delivers events as `text/event-stream`. Available for external services, multi-process architectures, and non-TypeScript consumers.

**Delivery Architecture**:

```
┌──────────────────────────────────────────────────────────────┐
│                        tap_core (Rust)                        │
│                                                               │
│  ┌──────────────┐                                             │
│  │ WAL Reader   │                                             │
│  │ & Decoder    │──────┐                                      │
│  └──────────────┘      │                                      │
│                        │                                      │
│  ┌─────────────────────▼─────────────────────────────────┐    │
│  │               Event Router                              │    │
│  │                                                         │    │
│  │  ┌──────────────────────┐  ┌────────────────────────┐  │    │
│  │  │ Channel A: SSE path  │  │ Channel B: Callback    │  │    │
│  │  │ → SSE Serializer     │  │ → napi-rs              │  │    │
│  │  │ → HTTP/2 Server      │  │   ThreadsafeFunction   │  │    │
│  │  │ → TCP Socket         │  │ → JS Event Loop        │  │    │
│  │  └──────────┬───────────┘  └───────────┬────────────┘  │    │
│  └─────────────┼──────────────────────────┼───────────────┘    │
└────────────────┼──────────────────────────┼────────────────────┘
                 │                          │
                 ▼                          ▼
        ┌──────────────────┐    ┌──────────────────────┐
        │ External         │    │ In-process JS        │
        │ EventSource      │    │ callback handler     │
        │ (HTTP clients)   │    │ tap.on('change',...) │
        └──────────────────┘    └──────────────────────┘
```

The SDK starts an embedded HTTP/2 server that emits events in the standard SSE format:

```
event: change
id: 0/16B37428:12345
data: {"op":"c","before":null,"after":{"id":1,"name":"Alice"},"source":{"db":"mydb","schema":"public","table":"users","lsn":"0/16B37428","tx_id":"12345","ts_ms":1717200000123},"ts_ms":1717200000150,"id":"0/16B37428:12345"}

event: change
id: 0/16B37430:12346
data: {"op":"u","before":{"id":1,"name":"Alice","email":"alice@old.com"},"after":{"id":1,"name":"Alice","email":"alice@new.com"},"source":{"db":"mydb","schema":"public","table":"users","lsn":"0/16B37430","tx_id":"12346","ts_ms":1717200000500},"ts_ms":1717200000530,"id":"0/16B37430:12346"}

event: heartbeat
data: {"ts_ms":1717200001000}

event: snapshot_complete
data: {"tables":["public.users","public.orders"],"total_rows":87654,"ts_ms":1717200002000}

event: error
data: {"code":"CONNECTION_LOST","message":"Connection to Postgres lost, reconnecting..."}
```

**SSE Event Types**:

| Event Type | Description | Frequency |
|---|---|---|
| `change` | A data change event (insert/update/delete/snapshot) | Per row change |
| `heartbeat` | Keep-alive ping (when no events for 10s) | Every 10s idle |
| `snapshot_start` | Snapshot phase began | Once per capture |
| `snapshot_progress` | Snapshot progress update | Per table chunk |
| `snapshot_complete` | Snapshot phase finished | Once per capture |
| `streaming_start` | Streaming phase began | Once per capture |
| `error` | Error information | On error |
| `shutdown` | Graceful shutdown notification | Once on stop |

**Consumer Example**:

```typescript
// Standard EventSource (browser, Node.js eventsource package)
const events = new EventSource("http://127.0.0.1:9911/events");

events.addEventListener("change", (e) => {
  const event: ChangeEvent = JSON.parse(e.data);
  console.log(event);
});

events.addEventListener("snapshot_complete", (e) => {
  console.log("Snapshot done:", e.data);
});

events.addEventListener("error", (e) => {
  console.error("SSE error:", e);
});
```

## 5. Data Flow — Step-by-Step Walkthrough

### End-to-End: `tap capture` → SSE → SDK `on('change')`

```
Step 1: CLI Startup
┌──────────────────────────────────────────────────────────────┐
│ tap capture --config .tap/config.toml                        │
│                                                              │
│ 1.1 Parse CLI args, load .tap/config.toml                    │
│ 1.2 Initialize tracing/logging subscriber                    │
│ 1.3 Open/initialize SQLite state store (.tap/state.db)       │
│     ├── Run CREATE TABLE IF NOT EXISTS for all tables        │
│     └── Read last checkpoint (lsn, sequence_number)          │
│ 1.4 Bind SSE HTTP/2 server to configured host:port           │
│     └── GET /events → SSE stream                             │
└──────────────────────────────────────────────────────────────┘

Step 2: Postgres Connection
┌──────────────────────────────────────────────────────────────┐
│ 2.1 tokio-postgres connects: replication=database            │
│ 2.2 Validate slot 'tap_slot' exists, or create it            │
│     └── CREATE_REPLICATION_SLOT tap_slot LOGICAL pgoutput    │
│ 2.3 Validate publication 'tap_publication' exists            │
│     └── CREATE PUBLICATION tap_publication FOR ALL TABLES    │
│ 2.4 If tables are specified, validate they exist             │
│ 2.5 If tables specified, ALTER PUBLICATION to include them   │
└──────────────────────────────────────────────────────────────┘

Step 3: Checkpoint Decision
┌──────────────────────────────────────────────────────────────┐
│ 3.1 Read offsets table: SELECT MAX(committed_lsn) FROM offsets│
│     │                                                         │
│     ├── If NULL (first run):                                  │
│     │   → Go to Step 4 (Snapshot)                             │
│     │   → Export snapshot at current LSN                      │
│     │                                                         │
│     └── If non-NULL (resume):                                 │
│         → Go to Step 6 directly (Streaming)                   │
│         → START_REPLICATION at saved LSN                      │
└──────────────────────────────────────────────────────────────┘

Step 4: Snapshot Phase
┌──────────────────────────────────────────────────────────────┐
│ 4.1 BEGIN READ ONLY ISOLATION LEVEL REPEATABLE READ          │
│ 4.2 SELECT pg_export_snapshot() → snapshot_id                │
│ 4.3 Record current WAL LSN (for transition to streaming)     │
│ 4.4 For each configured table (ordered by schema+name):       │
│     │                                                         │
│     ├── 4.4.1 SET TRANSACTION SNAPSHOT snapshot_id            │
│     ├── 4.4.2 SELECT * FROM table ORDER BY pk ASC             │
│     ├── 4.4.3 For each row:                                    │
│     │     ├── Build ChangeEvent { op: "r", after: row,        │
│     │     │     source: { snapshot: true, ... } }             │
│     │     ├── Push event to SSE channel                       │
│     │     └── Increment snapshot_row_count                   │
│     ├── 4.4.4 Record in snapshots table:                      │
│     │     INSERT INTO snapshots                               │
│     │       (table_name, snapshot_id, rows_count, status)     │
│     │     VALUES ('public.users', '...', 12345, 'completed')  │
│     └── 4.4.5 Emit snapshot_progress SSE event                │
│ 4.5 Emit snapshot_complete SSE event                          │
│ 4.6 Record snapshot_export_lsn in state store                 │
│ 4.7 COMMIT                                                    │
└──────────────────────────────────────────────────────────────┘

Step 5: Transition (Snapshot → Streaming)
┌──────────────────────────────────────────────────────────────┐
│ 5.1 Close snapshot transaction                               │
│ 5.2 Begin streaming at snapshot_export_lsn                   │
│ 5.3 Emit streaming_start SSE event                           │
└──────────────────────────────────────────────────────────────┘

Step 6: Streaming Phase
┌──────────────────────────────────────────────────────────────┐
│ 6.1 START_REPLICATION SLOT tap_slot LOGICAL pgoutput         │
│     0/16B37000 (the snapshot LSN or last checkpoint)          │
│                                                              │
│ 6.2 For each WAL message:                                     │
│     │                                                         │
│     ├── Relation message: cache column metadata               │
│     │   → Store in schema_cache (HashMap<RelationId, Schema>)│
│     │                                                         │
│     ├── Begin message: start new transaction context           │
│     │   → Record xid, begin_lsn, begin_timestamp              │
│     │                                                         │
│     ├── Insert message:                                       │
│     │   → Resolve relation_id to table schema                 │
│     │   → Build ChangeEvent { op: "c", after: new_row }       │
│     │   → Push to SSE channel                                 │
│     │                                                         │
│     ├── Update message:                                       │
│     │   → Build ChangeEvent { op: "u", before: old_row,       │
│     │     after: new_row }                                    │
│     │   → Push to SSE channel                                 │
│     │                                                         │
│     ├── Delete message:                                       │
│     │   → Build ChangeEvent { op: "d", before: old_row }      │
│     │   → Push to SSE channel                                 │
│     │                                                         │
│     ├── Commit message:                                        │
│     │   → Build event_id = "{commit_lsn}:{xid}"               │
│     │   → Flush checkpoint:                                    │
│     │     INSERT INTO offsets (committed_lsn, tx_id,           │
│     │       ts_ms) VALUES (commit_lsn, xid, now())             │
│     │   → Update in-memory metrics                             │
│     │   → Emit status update to status channel                 │
│     │                                                         │
│     └── Any unknown message type: skip with debug log         │
│                                                              │
│ 6.3 Memory monitoring: if buffer > max_buffer_size → pause    │
│ 6.4 Periodic heartbeat: if no events for 10s → emit heartbeat │
│     SSE event                                                 │
└──────────────────────────────────────────────────────────────┘

Step 7: SSE Delivery
┌──────────────────────────────────────────────────────────────┐
│ 7.1 HTTP/2 GET /events → 200 OK                               │
│     Headers: Content-Type: text/event-stream                   │
│              Cache-Control: no-cache                           │
│              Connection: keep-alive                           │
│                                                              │
│ 7.2 Event serialization:                                      │
│     event: change\n                                           │
│     id: 0/16B37428:12345\n                                    │
│     data: {"op":"c",...}\n\n                                  │
│                                                              │
│ 7.3 EventSource client receives and parses                     │
└──────────────────────────────────────────────────────────────┘

Step 8: SDK Callback (Dual Delivery)
┌──────────────────────────────────────────────────────────────┐
│ 8.1 Event Router receives ChangeEvent from Channel B           │
│ 8.2 napi-rs ThreadsafeFunction delivers event to:              │
│     ├── 8.2a In-process path (primary):                        │
│     │     ThreadsafeFunction queues callback on JS event loop  │
│     │     → tap.on('change', handler)(event)                   │
│     │     → 0.5–2ms latency, no serialization overhead         │
│     │                                                          │
│     └── 8.2b External SSE path (parallel):                     │
│           SSE serializer → HTTP/2 response → TCP socket        │
│           → External consumers (browser, other services)       │
│                                                               │
│ 8.3 If event ID already processed (dedup check): skip          │
│ 8.4 Otherwise: handler processes event (e.g., write to DB,     │
│     update cache, publish to queue)                           │
└──────────────────────────────────────────────────────────────┘

Step 9: Graceful Shutdown
┌──────────────────────────────────────────────────────────────┐
│ 9.1 SIGINT/SIGTERM received (Ctrl+C or kill)                  │
│ 9.2 Set shutdown_flag = true                                  │
│ 9.3 WAL reader loop checks flag after current message         │
│ 9.4 Flush final checkpoint:                                   │
│     INSERT INTO offsets (committed_lsn, tx_id, ts_ms)         │
│ 9.5 Emit shutdown SSE event                                   │
│ 9.6 Close SSE HTTP server (drain pending, 5s timeout)         │
│ 9.7 Send Standby status update to Postgres (Flush LSN)        │
│ 9.8 Close replication slot (if configured to drop on stop)    │
│ 9.9 Close Postgres connection                                 │
│ 9.10 Close SQLite state store                                  │
│ 9.11 Log: "Capture stopped. N events processed, 0 lost."      │
│ 9.12 Exit 0                                                   │
└──────────────────────────────────────────────────────────────┘
```

### 6.3 SSE Protocol Reference

**Endpoints**:

#### `GET /events` — SSE event stream

**Response Headers**:

```
HTTP/2 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
X-Tap-Version: 0.1.0
X-Tap-Instance: <uuid>
```

**Event Stream Format** (standard SSE):

```
event: <type>
id: <event-id>
data: <json-payload>

```

**Event Types** (see Section 4.3.3 for full descriptions):

| Event Type | Data Shape | Purpose |
|---|---|---|
| `change` | `ChangeEvent` (full JSON) | Data change |
| `heartbeat` | `{"ts_ms": number}` | Keep-alive |
| `snapshot_start` | `{"tables": string[], "timestamp": number}` | Snapshot started |
| `snapshot_progress` | `{"table": string, "rows": number, "total": number}` | Progress update |
| `snapshot_complete` | `{"tables": string[], "total_rows": number}` | Snapshot done |
| `streaming_start` | `{"lsn": string, "timestamp": number}` | Streaming began |
| `error` | `{"code": string, "message": string}` | Error notification |
| `shutdown` | `{"reason": string, "events_processed": number}` | Server shutting down |

**Replay / Resumption**:

The `id` field in SSE events carries the deterministic event ID. SSE-compatible clients can use the `Last-Event-ID` header on reconnection:

```
GET /events
Last-Event-ID: 0/16B37428:12345
```

Tap's SSE server resumes from the **nearest preceding commit boundary** rather than the exact event position, because the state store tracks commit LSNs (not individual event positions). Consumers should expect a few re-delivered events on reconnect and deduplicate by event ID. This provides basic replay without requiring a separate API.

### 6.4 Config Reference

**File**: `.tap/config.toml` (TOML format)

**Naming Convention**: The TOML config uses `snake_case` (Rust convention), while the `TapConfig` TypeScript interface uses `camelCase`. The mapping is straightforward: `slot_name` ↔ `slotName`, `max_batch_size` ↔ `maxBatchSize`, `flush_interval_ms` ↔ `flushIntervalMs`, etc.

```toml

type = "postgres"
connection = ""              # (required) Postgres connection string
slot_name = "tap_slot"       # Replication slot name
publication = "tap_publication"  # Publication name
plugin = "pgoutput"          # "pgoutput" or "wal2json"
tables = []                  # Empty = all tables in publication

[state]
backend = "sqlite"           # Only option in v0.1.0

[sink]
type = "sse"                 # Only option in v0.1.0
host = "127.0.0.1"           # Bind address
port = 0                     # 0 = random available port

[capture]
max_batch_size = 1000        # Events before forced checkpoint
flush_interval_ms = 1000     # Max ms between checkpoints
max_buffer_size = 10000      # Max buffered events (backpressure threshold)

[snapshot]
enabled = true               # Enable snapshot on first run
parallel = false             # v0.1.0: must be false

[reconnect]
max_retries = 10             # Connection retry count
initial_backoff_ms = 1000    # First retry delay
max_backoff_ms = 60000       # Maximum retry delay
jitter = true                # Add random ±20% jitter

[logging]
level = "info"               # trace, debug, info, warn, error
format = "pretty"            # "pretty" or "json"
```

## 7. Database Schema (SQLite State Store)

**File**: `.tap/state.db` (auto-created, user none exists)

### 7.1 DDL

```sql
-- Enable WAL mode for concurrent reads and better write performance
PRAGMA journal_mode=WAL;

-- Schema version tracking for auto-migration
CREATE TABLE IF NOT EXISTS schema_version (
    version    INTEGER NOT NULL PRIMARY KEY,
    applied_at TEXT    NOT NULL DEFAULT (datetime('now'))
);

INSERT INTO schema_version (version) VALUES (1);

-- ============================================================
-- LSN Checkpoints (core offset tracking)
-- ============================================================
CREATE TABLE IF NOT EXISTS offsets (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    -- The confirmed-flush LSN (e.g., "0/16B37428")
    committed_lsn   TEXT    NOT NULL UNIQUE,
    -- Postgres transaction ID
    tx_id           TEXT    NOT NULL,
    -- Wall clock when checkpoint was written
    ts_ms           INTEGER NOT NULL,
    -- Sequence number (monotonically increasing)
    sequence        INTEGER NOT NULL,
    -- Whether this was a final checkpoint on shutdown
    is_final        INTEGER NOT NULL DEFAULT 0,
    -- Capture instance metadata
    instance_id     TEXT,
    created_at      TEXT    NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_offsets_lsn ON offsets(committed_lsn);
CREATE INDEX idx_offsets_sequence ON offsets(sequence);
CREATE INDEX idx_offsets_final ON offsets(is_final) WHERE is_final = 1;

-- ============================================================
-- Snapshot Progress
-- ============================================================
CREATE TABLE IF NOT EXISTS snapshots (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    -- Fully qualified table name (e.g., "public.users")
    table_name      TEXT    NOT NULL,
    -- Snapshot identifier from pg_export_snapshot()
    snapshot_id     TEXT    NOT NULL,
    -- Number of rows processed
    rows_count      INTEGER NOT NULL DEFAULT 0,
    -- Status: 'in_progress', 'completed', 'failed'
    status          TEXT    NOT NULL DEFAULT 'in_progress',
    -- Timestamp when snapshot started
    started_at      TEXT    NOT NULL DEFAULT (datetime('now')),
    -- Timestamp when snapshot completed/failed
    completed_at    TEXT,
    -- Error message if failed
    error_message   TEXT,
    -- LSN at which this snapshot was taken
    snapshot_lsn    TEXT    NOT NULL,
    UNIQUE(table_name, snapshot_id)
);

CREATE INDEX idx_snapshots_table ON snapshots(table_name);
CREATE INDEX idx_snapshots_status ON snapshots(status);

-- ============================================================
-- Cached Schema Metadata
-- ============================================================
CREATE TABLE IF NOT EXISTS schema_cache (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    -- Fully qualified table name (e.g., "public.users")
    table_name      TEXT    NOT NULL UNIQUE,
    -- Column definitions as JSON array
    columns_json    TEXT    NOT NULL,
    -- Array of primary key column names
    primary_keys    TEXT    NOT NULL DEFAULT '[]',
    -- Postgres OID for the relation
    relation_oid    INTEGER,
    -- Schema version (incremented on DDL detection)
    schema_version  INTEGER NOT NULL DEFAULT 1,
    -- Last time schema was validated
    last_validated  TEXT    NOT NULL DEFAULT (datetime('now')),
    -- MD5 hash of the schema for quick comparison
    schema_hash     TEXT    NOT NULL,
    created_at      TEXT    NOT NULL DEFAULT (datetime('now')),
    updated_at      TEXT    NOT NULL DEFAULT (datetime('now'))
);

-- ============================================================
-- Skipped LSNs (for auditing decoding failures)
-- ============================================================
CREATE TABLE IF NOT EXISTS skipped_lsns (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    -- The LSN that failed to decode
    lsn             TEXT    NOT NULL,
    -- Transaction ID
    tx_id           TEXT,
    -- Error message
    error_message   TEXT    NOT NULL,
    -- Record details if available (JSON)
    raw_data        TEXT,
    created_at      TEXT    NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_skipped_lsns_lsn ON skipped_lsns(lsn);

-- ============================================================
-- Instance Metadata
-- ============================================================
CREATE TABLE IF NOT EXISTS instance_info (
    key             TEXT    PRIMARY KEY,
    value           TEXT    NOT NULL,
    updated_at      TEXT    NOT NULL DEFAULT (datetime('now'))
);

-- Seed instance info
INSERT OR IGNORE INTO instance_info (key, value) VALUES
    ('tap_version', '0.1.0'),
    ('created_at', datetime('now')),
    ('last_started', datetime('now'));
```

### 7.2 Table Descriptions

| Table | Purpose | Size Estimate |
|---|---|---|
| `offsets` | LSN checkpoint history. Each commit boundary produces one row. Retain last N checkpoints for recovery. | ~100 bytes/row, ~1M rows/year for busy DB → ~100MB/year |
| `snapshots` | Per-table snapshot progress. One row per table per snapshot run. Rows are immutable after completion. | ~200 bytes/row, negligible |
| `schema_cache` | Cached Postgres table schemas. One row per table. Invalidated when DDL detected. | ~1KB/table, negligible |
| `skipped_lsns` | Audit trail of WAL records that failed to decode. | Variable, expected near-zero |
| `instance_info` | Key-value metadata about the capture instance. | < 1KB |
| `schema_version` | Migration tracking for state store schema changes. | ~20 bytes/row |

### 7.3 Migration Strategy

- `schema_version` table holds the current state store schema version.
- On first load, the state backend checks version and runs migrations sequentially.
- v0.1.0 starts at version 1.
- Future versions will add migration functions (e.g., `migrate_v1_to_v2()`).

### 7.4 Backup & Recovery

- On state store open, copy to `.tap/state.db.bak` if no backup exists from this session.
- On checksum error (SQLite `PRAGMA integrity_check`), attempt restore from `.bak`.
- If both corrupt, fail with error guiding user to reinitialize (which triggers fresh snapshot).

## 8. Error Handling & Recovery

### 8.1 Connection Loss to Postgres

```
[Postgres connection lost]
        │
        ▼
┌───────────────────┐
│ Detect: WAL       │
│ stream timeout or │
│ TCP disconnect    │
└────────┬──────────┘
         │
         ▼
┌──────────────────────────────────────┐
│ Reconnection Loop                    │
│                                      │
│ for attempt in 1..=max_retries {     │
│     delay = min(initial_backoff *    │
│              2^(attempt-1), max)     │
│     delay += random(-jitter, +jitter)│
│     sleep(delay)                     │
│                                      │
│     match connect(connection_string) │
│         Ok(client) → resume_stream() │
│         Err(e) → log_warn(e)         │
│ }                                    │
│                                      │
│ max_retries exceeded → fatal_error() │
└──────────────────────────────────────┘
```

- On reconnect success, `START_REPLICATION` resumes from last persisted LSN in `offsets` table.
- No data loss: events already sent but not checkpointed are re-sent (idempotent consumers deduplicate via event ID).
- **Consumer responsibility**: Tap provides deterministic event IDs for deduplication. Consumers should store processed event IDs and skip duplicates. A built-in `TapEventStore` helper will be provided in a future SDK version.

### 8.2 Replication Slot Lost

Symptom: `ERROR: replication slot "tap_slot" does not exist`

```
Recovery:
1. Log fatal error with slot name
2. Preserve state store (do not corrupt offsets)
3. Exit with code 6
4. User instruction: run `tap init --force` to recreate slot + publication
5. On re-init, a full snapshot is triggered (offsets cleared)
```

### 8.3 Corrupt State Store

```rust
// On startup:
let integrity = conn.query_row("PRAGMA integrity_check", [], |row| row.get(0))?;
if integrity != "ok" {
    warn!("SQLite integrity check failed. Attempting backup restore...");
    std::fs::copy(".tap/state.db.bak", ".tap/state.db")?;
    let retry = conn.query_row("PRAGMA integrity_check", [], |row| row.get(0))?;
    if retry != "ok" {
        error!("Backup also corrupt. Manual recovery required.");
        info!("Run: tap init --force to recreate state store and re-snapshot.");
        process::exit(7);
    }
}
```

### 8.4 Graceful Shutdown Sequence

```
SIGINT/SIGTERM received
        │
        ▼
┌─────────────────────────────┐
│ 1. Set shutdown flag       │
│    WAL reader loop sees    │
│    flag on next iteration  │
└──────────┬─────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ 2. Finish current WAL msg  │
│    (do not abort mid-msg)  │
└──────────┬─────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ 3. Flush final checkpoint   │
│    INSERT INTO offsets      │
│    (committed_lsn, tx_id,   │
│     ts_ms, is_final=1)      │
└──────────┬─────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ 4. Emit shutdown SSE event  │
│    Close SSE HTTP server    │
│    (drain pending writes)   │
│    timeout: 5 seconds       │
└──────────┬─────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ 5. Send Standby status to   │
│    Postgres                 │
│    (confirm flush LSN)      │
└──────────┬─────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ 6. Close replication slot   │
│    (if config.drop_on_stop) │
└──────────┬─────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ 7. Close Postgres conn      │
│    Close SQLite conn        │
└──────────┬─────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ 8. Log summary:             │
│    "Capture stopped.        │
│     N events processed,     │
│     0 lost."                │
│    Exit 0                   │
└─────────────────────────────┘
```

**Total shutdown time target**: ≤ 1 second.

### 8.5 Edge Cases

| Edge Case | Handling |
|---|---|
| **Empty table during snapshot** | Zero `op: "r"` events emitted. Record as completed in `snapshots` table. |
| **Table without primary key** | Snapshot still runs (sequential is fine), but logged as warning since partial snapshot resume is not possible. Future versions use `ctid` for checkpoint. |
| **Schema changes during capture** | DDL detected via WAL. The `schema_cache` row is invalidated. New column mappings are fetched on next relation message. Column count mismatches during decoding produce a warning, and the event is emitted with available data. |
| **ALTER TABLE ADD COLUMN** | Old events may not include the new column. Events after the DDL include it. The SDK receives a mix. Consumers must handle optional fields (already `null`-able in generated types). |
| **DROP TABLE during capture** | WAL events reference the table's OID. The relation message for OID removal triggers a warning. Events for the dropped table are still emitted until the replication slot advances past the drop. |
| **Very large rows (> 1MB)** | Streamed as-is. No chunking in v0.1.0. Warning logged. Future versions will support configurable max event size with chunking. |
| **Network partition (long)** | Reconnection with exponential backoff for up to `max_retries`. After that, exit. Partition may cause the replication slot to advance past the disconnected consumer's LSN — events would be lost (this is a known Postgres limitation; `slot_keep` config in future). |
| **Concurrent `tap capture` instances** | SQLite state store uses WAL mode for concurrent reads. Only one capture instance should write. If a second instance starts, it will detect a lock and exit with a clear error. |

## 10. Security Considerations

### 10.1 Connection String Handling

- The Postgres connection string contains credentials (`postgres://user:password@host:port/db`).
- The connection string is stored in `.tap/config.toml` in plaintext.
- The `.tap/` directory should be added to `.gitignore` (auto-generated on `tap init`).
- Future versions will support environment variable interpolation (`${DB_PASSWORD}`) and secret manager integration.

### 10.2 No TLS in v0.1.0

- The SSE server binds to `127.0.0.1` by default (localhost-only).
- No transport encryption (TLS) for SSE or Postgres connections in v0.1.0.
- **Limitation**: Events flow in plaintext over localhost. Not suitable for cross-network capture.
- TLS will be added in v0.6.0 for all transports.

### 10.3 SQLite File Permissions

- The state store at `.tap/state.db` contains LSN positions but does not contain user data.
- Default permissions: `0600` (owner read/write only).
- The file is created with restrictive umask (`0o077`).

### 10.4 Replication Slot Security

- The Postgres user used for Tap requires `REPLICATION` attribute or `pg_create_physical_replication_slot` / `pg_create_logical_replication_slot` privileges.
- The replication slot consumes WAL on the Postgres server. An abandoned slot can cause WAL accumulation and disk exhaustion.
- Tap includes a slot retention guard: on graceful shutdown, the slot is preserved (not dropped) by default, allowing resumption. However, if the slot will not be reused, the user should configure `drop_on_stop = true`.
- Future versions will include a slot age check and alert.

### 10.5 Secrets in Logs

- Connection strings are redacted in log output (password masked: `postgres://user:***@localhost:5432/db`).
- Event payloads (`before`/`after` fields) are NOT redacted. Users should configure logging level to avoid logging event data in production.

## 12. Open Questions

| # | Question | Context | Decision / Status |
|---|---|---|---|
| 1 | **Default `drop_on_stop` behavior** | Should `tap capture` drop the replication slot on shutdown or retain it? Retaining enables resume but risks WAL bloat if abandoned. Dropping requires re-snapshot on next start. | **Retain by default** (resume UX better). Config flag `drop_on_stop` for cleanup. |
| 2 | **LSN flush strategy** | Flush on every commit (safest, but more writes) or batch (better perf, potential re-processing on crash)? | **Batch flush** — every N events or M milliseconds. Default: 1000 events or 1000ms, whichever comes first. Provides idempotent delivery with deterministic event IDs. |
| 3 | **SSE `Last-Event-ID` resume granularity** | Many SSE clients assume `Last-Event-ID` represents a resumption point, but WAL checkpoint is per-commit (may cover multiple events). Gap handling? | **Nearest commit boundary**: server cannot map `Last-Event-ID` to an exact WAL position; it resumes from the earliest preceding commit. Consumers expect re-delivery on reconnect. |
| 4 | **Binary size of napi-rs addon** | Rust + napi-rs native addon bundles libc, openssl, postgres client libs. Stripped binary may exceed npm size limits. | Need to measure. May require dynamic linking or `--no-default-features`. Investigate `strip = true` and `lto = "fat"`. |
| 5 | **macOS code signing for prebuilts** | macOS requires notarization for native binaries to avoid security warnings. | Notarization flow documented but not automated in v0.1.0. Linux has no such requirement. |
| 6 | **`wal2json` vs `pgoutput` for v0.1.0** | `pgoutput` is built-in (pg 14+), but `wal2json` is more widely deployed. Should both be first-class? | **`pgoutput` first-class, `wal2json` experimental**. `wal2json` does not provide `before` values for UPDATE/DELETE, has reduced test coverage, and requires extension install. See Section 4.1.1 for details. |
| 7 | **Snapshot progress reporting detail** | How frequently to report progress? Per row? Per 1000 rows? Per table? | **Per table**: report total row count on completion. Per-chunk progress (every 10,000 rows) via `tap capture` progress bar. |
| 8 | **Concurrent `tap capture` instances** | What happens if two `tap capture` processes target the same slot? SQLite will have write contention. | **Decided**: See Section 8.5 — On start, acquire SQLite `BEGIN EXCLUSIVE` lock. If held, exit with "another instance is running". |
| 9 | **npm package name** | Scoped `@tap/sdk` would require reserving the `@tap` org on npm before first release. | **Resolved: published as unscoped `tap-cdc`** — no org reservation needed, no scoped-namespace conflict, ships immediately with a personal `NPM_TOKEN`. See v0.1.0 implementation plan §12 question 4. |
| 10 | **Minimum Postgres version** | Logical replication via `pgoutput` requires pg 14+ (improved in pg 14). `wal2json` works on pg 10+. | **Minimum: Postgres 14** for v0.1.0. CI tests against 14, 15, and 16. If `wal2json` is used, pg 12+ (testing coverage needed). |

### 12.1 Trade-offs Acknowledged

| Trade-off | Chosen Approach | Rationale |
|---|---|---|
| **Memory vs. latency** | Buffer up to 10k events before backpressure | Prevents OOM on burst. Allows configurable tradeoff. |
| **SQLite performance vs. simplicity** | SQLite with WAL mode, fsync enabled | SQLite handles up to 10k writes/sec easily. CDC state is tiny. Simplicity wins over custom storage engine. |
| **Single-threaded snapshot vs. speed** | Sequential (no parallelism) | Complexity of chunked parallel snapshots (tracking ranges, resuming partial chunks) deferred to v0.2.0. Sequential is correct, simple, and sufficient for tables under 100M rows. |
| **pgoutput vs. wal2json as default** | pgoutput | Built-in, no extension install, official Postgres future. wal2json is available as fallback. |
| **TOML vs. YAML for config** | TOML | Simpler, unambiguous, native Rust support (`toml` crate). YAML can be a future alternative. |

## 13. Glossary

| Term | Definition |
|---|---|
| **CDC** | Change Data Capture — the process of tracking and capturing changes made to a database in real time. |
| **WAL** | Write-Ahead Log — Postgres's transaction log that records all changes. Logical decoding reads the WAL in a structured format. |
| **LSN** | Log Sequence Number — a pointer to a position in the Postgres WAL. Used to track replication progress (e.g., `0/16B37428`). |
| **Replication Slot** | A Postgres object that ensures the WAL is retained until all consumers have processed it. Prevents WAL from being recycled prematurely. |
| **Publication** | A Postgres object that defines which tables are replicated. Filterable by table name (and in future, by row/column filters). |
| **pgoutput** | The built-in Postgres logical decoding output plugin (pg 14+). Decodes WAL into structured insert/update/delete/relation messages. |
| **wal2json** | A Postgres extension that decodes WAL into JSON format. Supported by many cloud providers. Not built-in. |
| **SSE** | Server-Sent Events — a standard HTTP protocol (`text/event-stream`) for server-to-client streaming. Simpler than WebSockets for unidirectional event feeds. |
| **napi-rs** | A Rust crate and toolchain for building native Node.js addons. Provides `#[napi]` attribute macros to expose Rust functions to JavaScript with zero-copy interop. |
| **moon** | A monorepo build tool (moonrepo.dev) with first-class support for polyglot projects (Rust, Node, Python, Go, etc.) and deterministic task graphs. |
| **ChangeEvent** | Tap's standardized event envelope: `{ op, before, after, source, ts_ms, id }`. Consistent across all source databases and sink transports. |
| **Idempotent delivery** | Delivery semantics where events carry deterministic IDs so consumers can safely deduplicate. The server provides checkpointed delivery, but consumers must implement deduplication (e.g., by tracking processed event IDs). Not server-side exactly-once — effectively-once with idempotent consumers. |
| **Deterministic Event ID** | An event identifier derived from immutable source data (LSN + XID) so the same logical change always produces the same ID. |
| **pg_export_snapshot()** | A Postgres function that creates a transaction-consistent snapshot identifier. Used to ensure snapshot and streaming phase observe the same database state. |
| **State Store** | The local SQLite database (`.tap/state.db`) that persists LSN checkpoints, snapshot progress, and schema cache. |
| **tokio-postgres** | The async Rust PostgreSQL client library. Supports replication protocol via `start_replication()`. |
| **libpq** | The C client library for PostgreSQL, on which `tokio-postgres` is built for connection management. |
| **Testcontainers** | A library for spinning up disposable Docker containers in tests (Postgres, Redis, etc.). Used for integration testing. |
| **Backpressure** | A flow control mechanism where the consumer signals the producer to slow down. In v0.1.0, implemented as buffer threshold pause. |
