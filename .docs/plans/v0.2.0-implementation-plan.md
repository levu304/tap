## 1. Overview

### 1.1 What This Plan Covers

This implementation plan breaks down Tap v0.2.0 "Polyglot" into **10 sequential phases**, each producing a testable, shippable increment. The plan is derived from the canonical tech spec (`.docs/specs/v0.2.0-tech-spec.md`, 1773 lines, status Final) and the existing v0.1.0 codebase.

v0.2.0 expands from v0.1.0 (Postgres-only, TS SDK, SSE) across four axes:

1. **MySQL support**: New binlog adapter using `mysql_async` crate with `binlog` feature
2. **Python SDK**: `pyo3` + `maturin` native extension with asyncio stream API + Pydantic model generation
3. **Sidecar mode**: Standalone `tap-sidecar` binary on `localhost:9911` (HTTP/SSE/WebSocket) + gRPC on `9912`
4. **Transform engine**: QuickJS WASM embedded in wasmtime — filter, map, mask transforms

### 1.3 Breaking Changes

Two breaking changes from v0.1.0:

1. **`SourceMetadata.lsn` nullable**: Previously always a non-null string. Now `null` for MySQL events. Consumers that assumed non-null `lsn` must update.
2. **`tap dev` defaults to sidecar mode**: Pass `--embedded` for v0.1.0 behavior.

Additive (non-breaking): `ChangeEvent.last_transform_evaluated_at` field (null when no transforms).

### Phase 2: SQLite Migration (v0.1.0 → v0.2.0)

**Goal**: Migrate the existing SQLite state store schema from v1 to v2 with backward-compatible resume. Rename `committed_lsn` → `position`, add `adapter` column, rename `skipped_lsns` → `skipped_positions`, add `snapshot_chunks` table. PID lock file invariant.

**Dependencies**: Phase 1 (scaffold — existing state module must compile).

#### Files Created/Modified

| File | Purpose |
|---|---|
| `crates/tap-core/src/state/migration.rs` | Add v1→v2 migration logic (backup, rename columns, add tables) |
| `crates/tap-core/src/state/store.rs` | Rename OffsetRecord.committed_lsn → position; update all CRUD methods; add PID lock file logic |
| `crates/tap-core/src/state/mod.rs` | Export new migration version constant |
| `crates/tap-core/src/error/reconnect.rs` | Add `TapError::MigrationFailed(String)` variant |

#### Key Decisions

1. **`committed_lsn` → `position`**: Both Postgres LSN strings and MySQL binlog coordinate strings share the same column. No semantic loss since both are opaque position strings.
2. **Backup-first migration**: `state.db` → `state.db.v0.1.0.bak` before any schema changes. On failure, error message points to backup for recovery.
3. **PID lock file invariant**: `.tap/state.db.lock` with `flock()` prevents dual-writer corruption. Stale PID → overwrite. Lock released on graceful shutdown or OS process termination.
4. **`UNIQUE(sequence)` added via index not constraint**: Avoids destructive `ALTER TABLE` on existing v0.1.0 tables. `CREATE UNIQUE INDEX IF NOT EXISTS idx_offsets_sequence_unique ON offsets(sequence)`.
5. **No downgrade migration**: v0.2.0 data is forward-compatible only. Downgrade requires restoring the v0.1.0 backup.
6. **Resume ordering uses `sequence` (not `MAX(position)`)**: `sequence` is the monotonically increasing ordering key regardless of adapter type.

```rust
// state/migration.rs
pub const SCHEMA_VERSION_V1: u32 = 1;
pub const SCHEMA_VERSION_V2: u32 = 2;

pub fn migrate_v1_to_v2(conn: &rusqlite::Connection) -> Result<(), TapError> {
    // 1. PRAGMA integrity_check
    // 2. CREATE TABLE IF NOT EXISTS backup (or copy to state.db.v0.1.0.bak)
    // 3. ALTER TABLE offsets RENAME COLUMN committed_lsn TO position
    // 4. ALTER TABLE offsets ADD COLUMN adapter TEXT NOT NULL DEFAULT 'postgres'
    // 5. ALTER TABLE skipped_lsns RENAME TO skipped_positions
    // 6. ALTER TABLE skipped_positions RENAME COLUMN lsn TO position
    // 7. CREATE TABLE IF NOT EXISTS snapshot_chunks (...)
    // 8. CREATE UNIQUE INDEX IF NOT EXISTS idx_offsets_sequence_unique ON offsets(sequence)
    // 9. INSERT INTO schema_version (version) VALUES (2)
}
```

#### Implementation Notes

- **SQLite `RENAME COLUMN`**: Available since SQLite 3.25.0 (bundled with `rusqlite` via `bundled` feature). The v0.1.0 schema uses `committed_lsn` — rename is safe.
- **Busy timeout after migration**: Ensure `PRAGMA busy_timeout = 5000` is set before any migration DDL runs.
- **PID lock file path**: Always alongside state database: `{state_db_path}.lock` (e.g., `.tap/state.db.lock`).
- **`integrity_check` on startup**: Run after migration (or on first open for fresh databases). Exit with FATAL if check fails.
- **Existing v0.1.0 data**: All `offsets` rows get `adapter='postgres'` default. All `skipped_lsns` rows get renamed to `skipped_positions`.

#### Key Decisions

1. **`mysql_async` v0.37 with `features = ["binlog"]`**: Not a dedicated binlog crate. `mysql_async` implements `COM_BINLOG_DUMP` handshake, event streaming, GTID, and checksum handling natively (per §14.1).
2. **Position tracking as serialized string**: `"mysql-bin.000123:4567890"` stored in `offsets.position`. No GTID support in v0.2.0.
3. **Schema resolution via `information_schema.columns`**: MySQL does not emit relation descriptors like Postgres's pgoutput. Cache per table; invalidate on `QUERY_EVENT` (DDL).
4. **Row size limit (16 MB default)**: Events exceeding `max_row_size_bytes` are dropped with WARN log and counter increment. Prevents memory exhaustion.
5. **Big-integer wire encoding as string**: `BIGINT UNSIGNED`, `DECIMAL`/`NUMERIC` always serialized as JSON strings on wire. Consumers use `BigInt()` / `Decimal` for exact parsing.
6. **Pre-flight checks**: binlog_format=ROW (FATAL), binary logging enabled (FATAL), binlog_row_image (WARN), binlog_checksum (WARN), retention period (WARN), MySQL version (FATAL if < 5.7).

#### Implementation Notes

- **`num_cpus` for connection pool sizing**: Default pool size = `min(num_cpus, max_connections)`. Configurable via `source.max_connections`.
- **Row size check**: Compare wire-encoded JSON byte length of `after`/`before` to `max_row_size_bytes` (default 16 MB). Drop with counter increment.
- **Binlog checksum handling**: MySQL binlog events include a CRC32 checksum footer by default since MySQL 5.6. The parser must strip (or verify) the 4-byte checksum. WARN if not checksum-aware.
- **Pre-flight checks order**: Version check first (most independent), then binary logging, then binlog_format, then binlog_row_image, then binlog_checksum, then privileges.
- **Connection loss recovery**: Same exponential backoff as Postgres (1s → 2s → ... → max 60s, 10 retries).
- **Event ID format**: `"mysql:{binlog_file}:{binlog_offset}:{tx_id}"` for streaming events. Snapshot events use `"snap:{schema}.{table}:{pk_value}"`.

### Phase 4: Parallel Snapshot Engine

**Goal**: Implement chunked parallel snapshotting by primary key range for both Postgres and MySQL. Postgres uses `pg_export_snapshot()`; MySQL uses `FLUSH TABLES WITH READ LOCK` + `START TRANSACTION WITH CONSISTENT SNAPSHOT`. Progress tracking via `snapshot_chunks` table.

**Dependencies**: Phase 2 (SQLite migration — `snapshot_chunks` table), Phase 3 (MySQL snapshot consistency protocol).

#### Files Created/Modified

| File | Purpose |
|---|---|
| `crates/tap-core/src/snapshot/parallel.rs` | `ParallelSnapshotRunner` — chunked multi-worker snapshot engine |
| `crates/tap-core/src/snapshot/mod.rs` | Re-export ParallelSnapshotRunner |
| `crates/tap-core/src/config.rs` | Add `max_snapshot_workers`, `chunk_size_rows`, `checkpoint_rows`, `snapshot_mode`, `snapshot_worker_startup_timeout` |
| `crates/tap-core/src/adapters/mysql/snapshot.rs` | MySQL snapshot protocol (FLUSH TABLES WITH READ LOCK + consistent snapshot) |

#### Key Decisions

1. **`pg_export_snapshot()` for Postgres**: No global lock. All workers share a single exported snapshot ID for consistent point-in-time view.
2. **`FLUSH TABLES WITH READ LOCK` for MySQL**: Global lock held only during worker startup (not data scan). Lock duration is ms-scale for local connections. Metric-backed via `tap_snapshot_lock_held_ms`.
3. **Worker-startup timeout (default 30s)**: Prevents unresponsive worker from holding MySQL global lock indefinitely. Aborts snapshot with error.
4. **UUID skew detection**: Pre-flight checks by sampling 10k rows. Emits WARN if stddev of chunk sizes exceeds 2x mean. For time-ordered UUIDs (UUIDv7), one worker may process >90% of rows.
5. **Chunk boundary pre-computation**: `SELECT MIN(pk), MAX(pk)` → divide range into N equal-width chunks. N = `min(num_cpus, max_snapshot_workers)`.
6. **`skip_transforms_on_snapshot` default true**: Filter and map transforms skipped for snapshot events; mask always runs (privacy-critical).

#### Key Rust Types Defined

```rust
// snapshot/parallel.rs
pub struct ParallelSnapshotRunner {
    config: SnapshotConfig,
    state: Arc<Mutex<StateStore>>,
    event_tx: tokio::sync::broadcast::Sender<ChangeEvent>,
    tables: Vec<String>,
    adapter: AdapterType,  // Postgres or MySQL
}

impl ParallelSnapshotRunner {
    pub fn new(
        config: SnapshotConfig,
        state: Arc<Mutex<StateStore>>,
        event_tx: tokio::sync::broadcast::Sender<ChangeEvent>,
        tables: Vec<String>,
        adapter: AdapterType,
    ) -> Self;

    pub async fn run(&mut self) -> Result<SnapshotResult, TapError> {
        // 1. For each table with integer/UUID PK:
        //    a. Query MIN(pk), MAX(pk)
        //    b. Divide range into N equal-width chunks
        //    c. Spawn N worker tasks
        //    d. Each worker reads own chunk → emits op="r" events
        //    e. Progress tracked in snapshot_chunks table
        // 2. For tables without suitable PK: fall back to sequential
    }
}

pub struct SnapshotChunk {
    pub chunk_index: u32,
    pub pk_start: String,
    pub pk_end: String,
    pub last_pk: Option<String>,
    pub rows_emitted: u64,
    pub completed: bool,
}
```

#### Implementation Notes

- **Drop runner before awaiting connection handles**: Per `snapshotrunner-deadlock` insight — when using `tokio-postgres`'s `connect_plain()` which returns `Client + JoinHandle`, drop the runner struct before `handle.await` to avoid tokio self-deadlock.
- **MySQL lock protocol**: Coordinator acquires `FLUSH TABLES WITH READ LOCK` → all workers issue `START TRANSACTION WITH CONSISTENT SNAPSHOT` → coordinator `UNLOCK TABLES`. Workers scan without lock held.
- **Chunk boundary types**: Integer PKs use numeric comparison. UUID PKs use native type comparison (Postgres `uuid`, MySQL `BINARY(16)`). Non-integer/non-UUID PKs fall back to sequential.
- **Resume from `snapshot_chunks`**: On restart, check `snapshot_chunks` table. Completed chunks skipped. Partial chunks resume from `last_pk`.
- **Counter metrics**: `tap_snapshot_lock_held_ms`, `tap_snapshot_progress_pct{table=...}`.

#### Key Decisions

1. **wasmtime for QuickJS WASM hosting** (per §14.2): QuickJS compiled to WASM, hosted inside wasmtime. ~6 MB total additional binary size (~3.6 MB sidecar, ~2.4 MB embedded). Epoch-based interruption for precise 100ms timeout enforcement.
2. **Compile-once-at-load bytecode model**: QuickJS bytecode compiled once when transform is loaded, not per event. This is the key performance prerequisite for §11.2 targets (< 1 ms per map transform).
3. **Filter/map fail-open, mask fail-closed** (per §14.7): Filter/map errors pass event through unmodified (safer default). Mask errors reject the event (privacy-critical).
4. **Mask strategies**: `redact` → `"***REDACTED***"`, `hash` → HMAC-SHA-256 with stable salt, `null` → JSON null. Salt stored in `tap.toml` (`mask_salt`), not ephemeral.
5. **100ms per-event timeout**: Enforced via wasmtime epoch-based interruption. Configurable via `transform_timeout_ms`.
6. **16 MB memory limit per context**: Configurable via `transform_memory_limit_mb`. Excess triggers fail-open for filter/map, fail-closed for mask.
7. **`swc_core` for TypeScript → JS transpilation**: Used at `tap transform validate` time. QuickJS runs ES2020 JavaScript only.
8. **Absent dot-path in mask**: Default behavior = fail-closed (event rejected). Configurable per-transform via `mask_missing_field = "skip"`.

```python
        await self._core.start(loop, self._queue)

            while True:
                event = await self._queue.get()
                if event is None:
                    break
                yield ChangeEvent.model_validate(event)
        except asyncio.CancelledError:
            raise
        finally:
            await self._core.stop()

    async def start(self): ...
    async def stop(self): ...
    async def pause(self): ...
    async def resume(self): ...
```

#### Implementation Notes

- **GIL discipline**: Release GIL during all Rust-side processing (event decode, transform, serialization). Acquire GIL only during `call_soon_threadsafe` call. Measure GIL hold times in CI to ensure < 100µs per event.
- **Shutdown under CancelledError**: `asyncio.CancelledError` re-raised after `stop()`. `KeyboardInterrupt` signals Tokio runtime shutdown; `stop()` joins thread before returning.
- **BIGINT UNSIGNED → Python int**: Python `int` has arbitrary precision, so wire string is losslessly converted. `@field_validator` on the Pydantic model accepts both `int` and `str`.
- **`SourceMetadata.schema` field**: Pydantic v2 reserves `schema` as a model config key. Use `Field(alias="schema", serialization_alias="schema")` with `populate_by_name=True`.
- **`maturin` build matrix**: CI builds all 5 platform wheels. Use `--manylinux` Docker images for Linux cross-compilation.

#### Key Decisions

1. **`tap dev` defaults to sidecar mode**: Starts `tap-sidecar` as child process, holds PID, forwards signals. `--embedded` for v0.1.0 behavior.
2. **Orphan handling**: If `tap dev` is killed, sidecar may survive. Next invocation checks `:9911` port; logs warning and exits.
3. **Loopback binding check**: Sidecar binds to `127.0.0.1` by default. `bind_address = "0.0.0.0"` triggers startup warning.
4. **Config file discovery order**: `--config <PATH>` > `.tap/config.toml` > `./tap.toml` > `TAP_CONFIG_PATH` env var. First found wins; no merging.
5. **Config value precedence**: CLI flags > env vars (`TAP_SECTION_KEY`) > config file > built-in defaults. Each level only overrides explicitly set keys.
6. **`tap transform` does not deploy**: Validates and previews only. Transforms activated through `[[transforms]]` in `tap.toml`.
7. **First-run warning**: First `tap dev` invocation logs notice about sidecar being default mode.
8. **Port-in-use error**: If `:9911` is already bound, exit with actionable error suggesting `--port` or `--embedded`.

## 6. Risk & Mitigation

### Risk 1: MySQL Binlog Position No Longer Available After Retention Expiry

| Severity | Likelihood | Impact |
|---|---|---|
| **High** | Medium | If MySQL binlog retention is short and capture is stopped for an extended period, the binlog file for the checkpointed position may be purged. Capture cannot resume. |

**Mitigation**:
- Pre-flight check: warn if `expire_logs_days < 7`
- On startup, verify binlog position still exists: `SHOW BINARY LOGS`
- If position is lost: log the missing position, suggest re-snapshot, exit
- Document: "Set MySQL binlog retention to at least 7 days for production"

### Risk 2: wasmtime Binary Size Adds ~6 MB to Distribution

| Severity | Likelihood | Impact |
|---|---|---|
| **Medium** | Certain | Combined wasmtime runtime + QuickJS WASM blob adds ~6 MB to binary sizes. SDK binaries may approach size limits. |

**Mitigation**:
- Apply `lto = "fat"`, `opt-level = "z"`, `strip = true` in release profiles
- Compress QuickJS WASM blob with `include_bytes!` + zstd decompression at load time
- Measure final binary sizes; if SDK > 25MB, consider optional transform dependency feature gate
- Document: "Transforms add ~6 MB to binary size. Can be disabled via `--no-default-features`"

### Risk 3: MySQL Global Write Lock Duration During Parallel Snapshot

| Severity | Likelihood | Impact |
|---|---|---|
| **Medium** | Medium | `FLUSH TABLES WITH READ LOCK` blocks all writes during worker connection setup. On high-latency links or many workers, lock duration may exceed write-latency SLO. |

**Mitigation**:
- Default `max_snapshot_workers = 4` keeps lock window < 200ms on typical connections
- `snapshot_worker_startup_timeout` (30s) prevents indefinite lock
- Metric-backed: `tap_snapshot_lock_held_ms` enables alerting
- Document: "Reduce `max_snapshot_workers` for strict write-latency SLAs"
- For >5s expected lock duration, use `snapshot_mode = "sequential"` as fallback

### Risk 4: Python asyncio Bridge Deadlock or Thread Safety

| Severity | Likelihood | Impact |
|---|---|---|
| **High** | Medium | Incorrect GIL handling or thread synchronization in the bespoke channel bridge could cause deadlocks or event loss. Most common failure mode in pyo3 projects. |

**Mitigation**:
- Keep GIL held duration to absolute minimum (~call_soon_threadsafe call only)
- Release GIL during all Rust-side event processing
- Bounded queue (10,000 default) prevents unbounded memory growth
- Test harness: start/stop 100 times, kill/restart in rapid succession
- Stress test: 100k events through the bridge with slow consumer
- Document: "If the Python event loop is blocked for > 30s, the bridge may drop events"

### Risk 5: wasmtime Epoch Interruption API Incompatibility with QuickJS WASM

| Severity | Likelihood | Impact |
|---|---|---|
| **Medium** | Low | QuickJS compiled to WASM may not support wasmtime epoch-based interruption instructions. Timeout enforcement may not work. |

**Mitigation**:
- Verify epoch interruption works with QuickJS WASM during Phase 5 implementation
- If epoch interruption unavailable, fall back to wall-clock timeout in Rust with a tokio timer that drops the wasmtime store
- Document: "Timeout enforcement uses wasmtime epoch-based interruption. If unavailable, falls back to tokio timer-based store drop."

### Risk 6: gRPC `max_message_size` Configuration Mismatch

| Severity | Likelihood | Impact |
|---|---|---|
| **Low** | Medium | If `max_row_size_bytes` is raised above 16 MB without also increasing `sidecar.grpc_max_message_size_bytes`, large events will fail at the gRPC sink. |

**Mitigation**:
- Log a warning at startup if `max_row_size_bytes > sidecar.grpc_max_message_size_bytes`
- Document the coupling explicitly in config schema
- Provide a validation check in `tap init` that warns about the mismatch

## 7. Open Questions & Decisions

All open questions from the spec have been resolved (see §14.1–§14.10 in tech spec). The following is a summary of the resolved decisions:

| # | Question | Resolution | Ref |
|---|---|---|---|
| 1 | **MySQL crate: `mysql_async` or dedicated binlog crate?** | `mysql_async` v0.37 with `features = ["binlog"]` | §14.1 |
| 2 | **QuickJS embedding: `wasmtime` or `quickjs-wasm-rs`?** | `wasmtime` — epoch interruption, actively maintained | §14.2 |
| 3 | **gRPC port: shared with SSE (9911) or dedicated?** | Dedicated port `9912` | §14.3 |
| 4 | **Python asyncio bridge: `pyo3-asyncio` or bespoke?** | Bespoke channel bridge (loop.call_soon_threadsafe) | §14.4 |
| 5 | **MySQL consistent snapshot locking** | Accepted with metric-backed observability; per-table lock deferred | §14.5 |
| 6 | **`snapshot` field: omitted or explicit null?** | `skip_serializing_if` retained (unchanged from v0.1.0) | §14.6 |
| 7 | **Transform fail-open vs. fail-closed?** | Filter/map fail-open, mask fail-closed | §14.7 |
| 8 | **Windows Python SDK support?** | Deferred indefinitely; revisit at 3+ native requests | §14.8 |
| 9 | **Unix socket default path: `/tmp/tap.sock` or `.tap/tap.sock`?** | `.tap/tap.sock` with umask 0700 | §14.9 |
| 10 | **TS SDK sidecar reconnection policy** | Exponential backoff (100ms→30s, ±20% jitter), resume via last_event_id, stop after 10 failures | §14.10 |

### Remaining Implementation Decisions

These decisions are left to the implementer:

| # | Decision | Options | Guidance | Affects |
|---|---|---|---|---|
