## 1. Overview

### 1.1 What This Plan Covers

This implementation plan breaks down Tap v0.1.0 "Foundation" into **11 sequential phases**, each producing a testable, shippable increment. The plan is derived from the canonical tech spec (`.docs/specs/v0.1.0-tech-spec.md`, 2680 lines), the roadmap (`.docs/ROADMAP.md`), and the brainstorming document (`.docs/BRAINSTORMING.md`).

### 1.2 Scope Recap

| What's In | What's Out |
|---|---|
| Postgres CDC via logical replication (pgoutput + wal2json experimental) | Sidecar mode (v0.2.0) |
| SQLite state store (6 tables, WAL mode) | Python SDK (v0.2.0) |
| ChangeEvent envelope (op, before, after, source, ts_ms, id) | Kafka/Redis/NATS sinks |
| Sequential snapshotting with `pg_export_snapshot()` | Parallel snapshots |
| HTTP/2 SSE sink (8 event types) | WebSocket, gRPC |
| Dual delivery: in-process callback + SSE | Schema registry |
| TS SDK via napi-rs (Tap class, 5 methods) | Cluster mode |
| CLI (5 commands: init, capture, inspect, dev, test) | TLS, auth, RBAC |
| moon monorepo + GitHub Actions CI | OpenTelemetry |
| Prebuilt binaries (5 targets) | Dashboard/UI |

### Phase 2: Core Types — ChangeEvent, Config, Error Types

**Goal**: Define all shared data types so every subsequent phase compiles against stable structs.

**Dependencies**: Phase 1 (scaffold + lib.rs with `pub mod` stubs).

| `crates/tap-core/src/lib.rs` | Wire modules: `pub mod event; pub mod error; pub mod config;` |

#### Key Decisions

1. **`op` field as `String` not enum**: Matches Debezium convention (`"c"`, `"u"`, `"d"`, `"r"`). An enum would require serialization mapping anyway. The `Operation` enum handles internal matching; serialized as string.
2. **`thiserror` for error types**: Lightweight derive macro for `std::error::Error` impls. Each error variant carries domain-specific context.
3. **Config deserialization with `serde`**: TOML uses `snake_case`; TS SDK uses `camelCase`. Struct fields use `#[serde(rename_all = "camelCase")]` on the Rust side so TOML stays idiomatic snake_case but the Rust struct matches TS conventions.
4. **`anyhow` for CLI, `thiserror` for library**: CLI binary uses `anyhow::Result` for ergonomic error handling. Library uses typed `thiserror` errors.

### Phase 3: Postgres Connection — Replication Slot & Publication

**Goal**: Establish and manage the Postgres connection, replication slot, and publication lifecycle.

**Dependencies**: Phase 1 (scaffold), Phase 2 (config + error types).

#### Files Created/Modified

| File | Purpose |
|---|---|
| `crates/tap-core/src/postgres/mod.rs` | Module re-exports |
| `crates/tap-core/src/postgres/connection.rs` | `PgConnection` — connect, slot management, publication management |
| `crates/tap-core/src/lib.rs` | Add `pub mod postgres;` |

#### Key Decisions

1. **`tokio-postgres` with `replication` mode**: Connection string must include `?replication=database`. Validate this on init and surface a clear error if missing.
2. **Slot creation via `CREATE_REPLICATION_SLOT`**: Use `client.simple_query()` for DDL statements. Slot names are validated (alphanumeric + underscore only).
3. **Publication defaults to `FOR ALL TABLES`**: If no tables are specified, create publication with `FOR ALL TABLES`. If tables are specified, use `FOR TABLE t1, t2, ...`.
4. **Connection lifecycle as a state machine**: `Disconnected → Connecting → Connected(validating) → Ready → Reconnecting → Disconnected`. Each transition is logged.
5. **Validator queries on startup**: Run `SELECT slot_name FROM pg_replication_slots` and `SELECT pubname FROM pg_publication` to check existing state before creating.

### Phase 4: WAL Decoding — pgoutput + wal2json

**Goal**: Decode logical replication messages into structured `ChangeEvent` instances. pgoutput is first-class; wal2json is experimental.

**Dependencies**: Phase 2 (ChangeEvent types), Phase 3 (PgConnection + ReplicationStream).

#### Files Created/Modified

| File | Purpose |
|---|---|
| `crates/tap-core/src/postgres/decoder.rs` | `WalDecoder` trait + `PgoutputDecoder` + `Wal2JsonDecoder` |
| `crates/tap-core/src/postgres/mod.rs` | Re-export decoder types |

#### Key Decisions

1. **Trait-based decoder**: `trait WalDecoder { type Input; fn decode(&self, msg: &[u8]) -> Result<Vec<ChangeEvent>>; }`. Allows swapping implementations at runtime.
2. **pgoutput decoding maps directly to pgoutput message types**: `Begin`, `Insert`, `Update`, `Delete`, `Commit`, `Relation`, `Type`, `Origin`. The `Relation` message populates a `HashMap<RelationId, RelationSchema>` cache.
3. **wal2json parsing is simpler but lossier**: wal2json emits per-transaction JSON blobs. No `before` values for UPDATE/DELETE. No relation messages. Schema must be resolved from the state store.
4. **LSN tracking on commit boundaries**: Only `Commit` messages trigger checkpoint writes. Single WAL record errors skip and log to `skipped_lsns` table.
5. **Schema cache**: `HashMap<RelationId, RelSchema>` populated by `Relation` messages in pgoutput; stored in state store for wal2json fallback.

#### Implementation Notes

- **pgoutput binary format** is well-documented in the Postgres source (`src/backend/replication/pgoutput/pgoutput.c`). The byte layout is:
  - Byte 1: message type (`b`=Begin, `i`=Insert, `u`=Update, `d`=Delete, `c`=Commit, `r`=Relation, `y`=Type, `o`=Origin)
  - Followed by message-specific fields (LSN, XID, tuple data, etc.)
  - Tuple data uses the `pg_logical_slot_get_changes` format: `n_columns, [flags, col_len, col_data]*`
- **wal2json JSON format** is a single JSON object per transaction with a `"change"` array. Need to handle the transaction framing carefully.
- **Malformed records**: `decode()` returns `TapError` with the LSN position. The caller catches this, logs it, and continues. The bad LSN is written to `skipped_lsns`.
- **Performance**: Tuple deserialization is the hot path. Use `Bytes` for zero-copy where possible.

### Phase 5: SQLite State Store

**Goal**: Persist LSN checkpoints, snapshot progress, schema cache, and instance metadata in a local SQLite database with WAL mode and migration support.

**Dependencies**: Phase 2 (config + error types).

#### Files Created/Modified

| File | Purpose |
|---|---|
| `crates/tap-core/src/state/mod.rs` | Module re-exports |
| `crates/tap-core/src/state/store.rs` | `StateStore` — 6 tables, CRUD operations |
| `crates/tap-core/src/state/migration.rs` | `migrate()` version check + sequential migrations |
| `crates/tap-core/src/lib.rs` | Add `pub mod state;` |

#### Key Decisions

1. **WAL mode always on**: `PRAGMA journal_mode=WAL` on every open. Required for concurrent read access during snapshot/writing.
2. **`rusqlite` with `bundled` feature**: Bundles SQLite so users don't need a system install. Adds ~2MB to binary size but guarantees version compatibility.
3. **Batch checkpointing**: Insert to `offsets` on every commit or batch boundary. Reads use `SELECT MAX(committed_lsn)` for resume.
4. **Backup before write**: On startup, `PRAGMA integrity_check` then copy to `state.db.bak`. On checksum mismatch, restore from backup.
5. **Exclusive lock on start**: `BEGIN EXCLUSIVE TRANSACTION` at startup. If lock held, exit with "another instance is running" error. Prevents dual-writer corruption.
6. **Migration strategy**: `schema_version` table tracks current version. `migrate()` checks version and runs forward-only SQL migrations sequentially. v0.1.0 starts at version 1 (the initial DDL).

#### Implementation Notes

- **`PRAGMA busy_timeout = 5000`**: Prevents immediate failure on lock contention. Waits 5 seconds.
- **`PRAGMA synchronous = NORMAL`**: Balances safety and speed. WAL mode with NORMAL is crash-safe.
- **Backup strategy**: Copy `state.db` → `state.db.bak` on first open. Only one backup file kept. The `.bak` file is overwritten each session.
- **Resume flow**: On startup, read `SELECT committed_lsn FROM offsets WHERE is_final=1 ORDER BY sequence DESC LIMIT 1`. If exists, resume from that LSN. If not, read `SELECT MAX(sequence), committed_lsn FROM offsets`. If neither, start fresh snapshot.
- **Concurrent access**: `StateStore` is not `Send + Sync` — it holds a `rusqlite::Connection` which is not thread-safe. Use `Mutex<StateStore>` in `Arc` for shared access, or keep it on one task and use message passing.

### Phase 6: Snapshot Engine — Sequential Snapshot

**Goal**: Perform a full-table sequential snapshot using `pg_export_snapshot()` for a consistent point-in-time view, emitting `op: "r"` events with progress tracking.

**Dependencies**: Phase 3 (PgConnection), Phase 4 (WAL decoding), Phase 5 (StateStore).

#### Files Created/Modified

| File | Purpose |
|---|---|
| `crates/tap-core/src/snapshot/mod.rs` | Module re-exports |
| `crates/tap-core/src/snapshot/runner.rs` | `SnapshotRunner` — sequential snapshot logic |
| `crates/tap-core/src/lib.rs` | Add `pub mod snapshot;` |

#### Key Decisions

1. **Sequential only**: Tables processed one at a time, ordered by schema + name. Parallelism deferred to v0.2.0.
2. **`pg_export_snapshot()` for consistency**: The snapshot and streaming phases share a single exported snapshot ID, guaranteeing no gap.
3. **`SET TRANSACTION SNAPSHOT` per table**: Each table scan opens a new transaction with the exported snapshot. This keeps individual transactions short and avoids long-running idle transactions.
4. **Primary key sort**: `SELECT * FROM table ORDER BY pk_col ASC`. Without a PK, log a warning, use `ORDER BY ctid`, but note that resume on failure is best-effort.
5. **Checkpoint on every 10,000 rows**: Progress saved to `snapshots` table. If interrupted, snapshot resumes from the last table that wasn't marked `completed`.

#### Rust Types Defined

```rust
// snapshot/runner.rs
pub struct SnapshotRunner {
    pg_conn: PgConnection,
    state: Arc<Mutex<StateStore>>,
    config: SnapshotConfig,
    event_tx: tokio::sync::mpsc::UnboundedSender<ChangeEvent>,
    tables: Vec<String>,
}

impl SnapshotRunner {
    pub fn new(
        pg_conn: PgConnection,
        state: Arc<Mutex<StateStore>>,
        config: SnapshotConfig,
        event_tx: tokio::sync::mpsc::UnboundedSender<ChangeEvent>,
        tables: Vec<String>,
    ) -> Self;
    
    pub async fn run(&mut self) -> Result<SnapshotResult, TapError> {
        // 1. BEGIN READ ONLY ISOLATION LEVEL REPEATABLE READ
        // 2. SELECT pg_export_snapshot() → snapshot_id
        // 3. Record snapshot_lsn from current WAL position
        // 4. For each table (sorted schema+name):
        //    a. SET TRANSACTION SNAPSHOT snapshot_id
        //    b. SELECT * FROM table ORDER BY pk
        //    c. Emit ChangeEvent { op: "r", snapshot: true } per row
        //    d. Every 10k rows: checkpoint to state store
        //    e. Mark table completed
        // 5. Emit snapshot_complete event
        // 6. Return SnapshotResult { snapshot_id, lsn, total_rows }
    }
}

pub struct SnapshotResult {
    pub snapshot_id: String,
    pub lsn: Lsn,
    pub total_rows: u64,
    pub tables_snapshot: Vec<String>,
}
```

#### Implementation Notes

- **Gap prevention**: The `snapshot_lsn` recorded in Step 3 is the LSN at snapshot export time. The streaming phase starts from this exact LSN, ensuring no events are missed between the end of snapshot and the start of streaming.
- **DDL during snapshot**: `pg_export_snapshot()` freezes the catalog. DDL occurring after export but before your table is scanned will produce errors. Catch `catalog snapshot does not reflect changes` errors and re-export.
- **Empty tables**: Zero rows scanned, marked as completed immediately. No events emitted.
- **Tables without PK**: Log a `warn!` with the table name. Use `ORDER BY ctid` for ordered scan. On interrupt, the table is fully re-scanned (cannot resume partial).
- **Large rows (>1MB)**: No chunking in v0.1.0. Warn via log. Stream as-is.
- **Progress bar output** (via `tracing` or direct stderr): `■ table [====> ] 42% 12,345 rows`. Track total rows for each table.

### Phase 7: SSE Server — HTTP/2 Event Stream

**Goal**: Deliver `ChangeEvent`s over HTTP/2 SSE with 8 event types, health/status endpoints, and Last-Event-ID resume logic.

**Dependencies**: Phase 2 (ChangeEvent), Phase 4 (decoder).

#### Key Decisions

1. **HTTP/2 only**: Use `hyper` (v1) with `http2` feature or `h2` crate directly. v0.1.0 does not support HTTP/1.x for SSE.
2. **`tokio::sync::broadcast` for fan-out**: Each SSE client gets a `broadcast::Receiver`. When WAL produces an event, it's broadcast to all connected clients.
3. **Buffer management**: `broadcast::channel(buffer_size)` with `max_buffer_size = 10,000`. Slow consumers cause `RecvError::Lagged`. Disconnected consumers detected via dropped `Receiver`.
4. **Health endpoints share the same server**: `GET /events` (SSE), `GET /health` (health check), `GET /status` (capture status). Use a single `hyper::Server` with route matching.
5. **Last-Event-ID resume**: Client sends `Last-Event-ID: 0/16B37428:12345`. Server reads this, extracts the LSN component, queries the state store for the nearest preceding commit boundary, and resumes from there. Note: this is **best-effort** — some event re-delivery is expected.
6. **Heartbeat timer**: If no events for 10 seconds, emit `event: heartbeat\ndata: {"ts_ms":...}\n\n`. Implemented as a `tokio::time::interval` task.


#### Implementation Notes

- **`hyper` server start**: Bind before any capture starts. The SSE server should be ready to accept connections immediately, even during snapshot phase.
- **`Last-Event-ID` parsing**: Extract the LSN from `{lsn}:{tx_id}` format. Query `SELECT committed_lsn FROM offsets WHERE committed_lsn <= ? ORDER BY sequence DESC LIMIT 1`. Resume from that LSN. This means some events near the reconnect boundary may be re-delivered.
- **Client disconnect**: `broadcast::Receiver` is dropped when the HTTP response body channel closes. Detect via `select!` on recv + shutdown signal.
- **Concurrent clients**: The broadcast channel fan-out means one slow client doesn't block others. The trade-off is memory: each broadcast message is cloned to every receiver.

#### napi-rs Async Bridge Architecture

```
WAL Reader (tokio task)
  │
  ├── tokio::sync::broadcast ──► SSE HTTP server (external consumers)
  │
  └── tokio::sync::mpsc (bounded, 10k) ──► napi::ThreadsafeFunction
                                              │
                                          JS Event Loop
                                              │
                                       tap.on('change', handler)
```

#### Implementation Notes

- **`max_buffer_size` overflow**: When Channel B is full (JS consumer slow), the WAL reader pauses. The same backpressure mechanism as SSE buffering. When the drain, the sender's `send()` future completes.
- **Error propagation**: Rust `TapError` → convert to `napi::Error` → deliver via `ThreadsafeFunction` to `on('error')` handler.
- **Binary size concerns**: `#[napi]` auto-generates JS glue code. Combined with Rust's `rusqlite` (bundled), the binary may approach 20MB. Mitigation: `lto = "fat"`, `opt-level = "z"`, `strip = true`, and exclude debug symbols.
- **Graceful shutdown from JS**: When `stop()` is called, set an `AtomicBool` shutdown flag, flush LSN, close connections, stop tokio runtime.
- **Runtime lifecycle**: The napi-rs addon starts a `tokio` runtime on first use. This runtime lives for the lifetime of the Node.js process (or until `stop()` is called). Use `#[napi]` `tokio::runtime::Runtime` stored in a `OnceCell`.

## 6. Risk & Mitigation

### Risk 1: napi-rs Binary Exceeds npm Size Limits

| Severity | Likelihood | Impact |
|---|---|---|
| **High** | Medium | npm rejects publish > ~50MB compressed. Users can't `npm install tap-cdc`. |

**Mitigation**:
- Set `lto = "fat"` and `opt-level = "z"` in release profile for `packages/sdk-ts`
- Set `strip = true` in `Cargo.toml` (strips debug symbols, saves 30-50%)
- Use musl target to avoid glibc bundling (`x86_64-unknown-linux-musl`)
- Measure binary size at Phase 9; if still > 20MB:
  - Split out `rusqlite` as a separate native addon (users install on demand)
  - Use `--no-default-features` to trim tokio/rusqlite features
  - Consider dynamic linking of bundled libs
- Target: < 15MB per platform (compressed), < 20MB uncompressed

### Risk 2: SQLite Write Throughput Bottleneck

| Severity | Likelihood | Impact |
|---|---|---|
| **Medium** | Low | On high-throughput databases (> 5k events/sec), per-commit SQLite writes block the WAL reader. |

**Mitigation**:
- WAL mode enabled by default (concurrent reads during writes)
- Batch checkpointing: write to SQLite every 1000 events OR 1000ms, not per-commit
- Use `PRAGMA synchronous = NORMAL` (not FULL) — crash-safe with WAL
- Consider `PRAGMA mmap_size` for larger state databases
- If bottleneck persists: collect batches in a `Vec` in memory, `INSERT` via prepared statement with batch binding
- Monitor via `tracing`: log time spent in SQLite writes. Alert if > 100ms per write.

### Risk 3: Abandoned Replication Slot Causes Postgres WAL Bloat

| Severity | Likelihood | Impact |
|---|---|---|
| **Critical** | Medium | Unmonitored abandoned slot prevents WAL recycling. Postgres fills disk. Database crashes. |

**Mitigation**:
- Default behavior: **retain slot on stop** (for resume), but add config flag `drop_on_stop = true`
- On startup, check `pg_replication_slots` for slot age:
  ```sql
  SELECT slot_name, pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn))
  FROM pg_replication_slots WHERE slot_name = 'tap_slot';
  ```
- Log slot age in `tap capture` output: `Slot: tap_slot (lag: 12ms)`
- In CI, test that `tap init --force` recreates slot + clears offsets
- Document: "Set up Postgres WAL monitoring. Abandoned slots are the #1 cause of Postgres disk issues."
- Future: `slot_max_lag_bytes` config with auto-drop when exceeded (v0.2.0)

### Risk 4: No TLS Until v0.6.0 Blocks Regulated Production Adoption

| Severity | Likelihood | Impact |
|---|---|---|
| **Medium** | High | Finance, healthcare, and enterprise users cannot adopt v0.1.0 due to compliance requirements. |

**Mitigation**:
- Document limitation prominently in README and quickstart:
  > "Tap v0.1.0 does not support TLS. The SSE server binds to 127.0.0.1 by default. Do not expose the SSE port to untrusted networks."
- SSE server binds to `127.0.0.1` by default (localhost-only)
- Connection string redaction in logs (password masked)
- SQLite file permissions set to `0600`
- Provide stunnel/nginx reverse proxy example for early adopters who need encryption
- Target: TLS in v0.6.0 (tracked in ROADMAP.md)

### Risk 5: Postgres Version Differences

| Severity | Likelihood | Impact |
|---|---|---|
| **Medium** | Medium | pgoutput message format differs between pg 14, 15, 16. wal2json compatibility varies. Replication permissions differ. |

**Mitigation**:
- CI matrix tests against pg 14, 15, and 16
- Pgoutput decoder tested against each version's known binary format
- Release note states minimum pg version: **Postgres 14+** (for pgoutput)
- wal2json disclaimer: experimental, tested against pg 14-16 with wal2json 2.6+
- Feature detection on connect:
  ```rust
  let version: String = client.query_one("SELECT version()", &[]).await?.get(0);
  // Parse "PostgreSQL 16.1 on x86_64..." to u16
  let pg_version = parse_pg_version(&version);
  ```
- If pg < 14, warn and force wal2json mode
- Monitor for breaking changes in pg 17 (logical decoding improvements)

### Risk 6: Race Condition on Dual Delivery Path

| Severity | Likelihood | Impact |
|---|---|---|
| **Medium** | Medium | SSE server and napi-rs callback receive events in different order due to separate channels. |

**Mitigation**:
- Both channels read from the same `ChangeEvent` source (single WAL reader task)
- The event router splits via `tokio::sync::broadcast` (fan-out, same order)
- SSE and callback both receive events in WAL commit order
- Document: "Events are delivered in LSN order. Both SSE and in-process callback maintain commit ordering."
- Test: verify ordering parity between channels

## 7. Open Questions & Decisions

These decisions must be made by the implementer before or during development.

| # | Question | Options | Recommendation | Affects |
|---|---|---|---|---|
| 1 | **Tokio runtime for snapshot: tokio task vs std::thread** | (a) Spawn snapshot on same tokio runtime with `tokio::task::spawn_blocking` (blocks a thread for row iteration); (b) Dedicated `std::thread` with its own sync Postgres connection | **(a) `spawn_blocking`** — simpler, shares the state store, one Postgres connection type. The snapshot is I/O-bound (SELECT), not CPU-bound, so a blocking tokio task is fine. | P6 |
| 2 | **Error crate: `thiserror` vs `snafu` vs custom** | (a) `thiserror` (derive macro, std Error trait); (b) `snafu` (context-rich, backtrace); (c) Custom `Error` enum | **(a) `thiserror`** — lighter than snafu, more idiomatic than custom. Use `#[from]` for automatic conversion. | P2 |
| 3 | **Testcontainers crate: `testcontainers` vs `testcontainers-modules`** | (a) `testcontainers` (community, generic); (b) `testcontainers-modules` (niche, fewer features) | **(a) `testcontainers`** — wider adoption, async support, Postgres module built-in. | P10 |
| 4 | **npm package name** | (a) `tap-cdc` (unscoped, no org needed) — chosen; (b) `@tap/sdk` (scoped, requires reserving the `@tap` npm org); (c) `@tapdev/sdk` (scoped fallback) | **(a) `tap-cdc`** — no org creation, ships immediately with a personal `NPM_TOKEN`. Per-platform packages follow napi-rs convention: `tap-cdc-darwin-arm64`, `tap-cdc-linux-x64-gnu`, etc. | P9 |
| 5 | **SSE HTTP library: `hyper` vs `axum` vs `warp`** | (a) `hyper` (lowest-level, manual routing); (b) `axum` (tokio-native, extractors); (c) `warp` (filter-based) | **(b) `axum`** — built on `hyper`, tokio-native, ergonomic routing, SSE via `axum::response::sse`. Hyper is too low-level; warp is less maintained. | P7 |
| 6 | **CLI TOML parsing: `toml` vs `toml_edit`** | (a) `toml` (read-only, serde); (b) `toml_edit` (round-trip preserving) | **(a) `toml`** — v0.1.0 only reads config, never modifies it. `toml_edit` needed when CLI writes config (v0.2.0+). | P2, P8 |
| 7 | **macOS code signing for prebuilt binaries** | (a) Skip signing (gatekeeper warning on first run); (b) Set up developer ID + notarization | **(a) Skip for v0.1.0** — document that macOS users need `xattr -d com.apple.quarantine tap` or right-click → Open. Notarization adds CI complexity. | P11 |
| 8 | **wal2json maturity: experimental vs remove** | (a) Keep as experimental with clear docs; (b) Remove, add back in v0.2.0 with more testing | **(a) Keep** — wal2json is widely deployed on managed Postgres (RDS, Cloud SQL). Clearly mark as experimental, reduced test coverage. | P4 |
| 9 | **pgoutput decoder: hand-written vs `pgoutput` crate** | (a) Hand-write binary parser (full control); (b) Use `pgoutput` crate (less code, dependency risk) | **(a) Hand-write** — the pgoutput binary format is simple (message type byte + fields). A dependency may lag Postgres releases. ~300 lines of safe Rust. | P4 |
| 10 | **Snapshot parallelism in v0.1.0** | (a) Sequential only (spec); (b) Add basic chunked parallelism as stretch goal | **(a) Sequential** — explicitly a non-goal. Parallel snapshots in v0.2.0. | P6 |

| Criteria | How to Verify | Phase |
|---|---|---|
| `tap init` scaffolds a working project | Run `tap init --db <connection>` → `.tap/config.toml` exists | P8 |
| `tap capture` captures row changes | INSERT → SSE event received with op="c" | P8 |
| Snapshot emits `op: "r"` events | 10k pre-populated rows → 10k events received | P6 |
| Graceful shutdown no data loss | Stop capture → resume → no gaps | P8 |
| SSE `Last-Event-ID` resume | Disconnect → reconnect with header → events resume | P7 |
| SDK `on('change')` fires | JS handler called on every insert | P9 |
| Deterministic event IDs | Same LSN+XID → same ID string | P2 |
| Pause/Resume works | `pause()` stops events → `resume()` restarts them | P9 |

### 8.2 Performance Requirements

| Criteria | Target | How to Verify |
|---|---|---|
| Single-row insert → callback | < 100ms p50 | `cargo bench` with tracing |
| Snapshot throughput | > 10k rows/sec | Measure 100k row table |
| Cold start | < 1s | `time tap capture` |
| Graceful shutdown | < 1s | SIGINT → process exit |
| Binary size | < 15MB | `ls -lh target/release/tap` |
| Memory idle | < 10MB RSS | `ps` measurement |
